## MAIN WORKFLOW FOR A FEATURE (порядок создания/заполнения артефактов для конкретной фичи)
1. Пользователь вручную создает папку agents/analytics/ba/features/feature_'порядковый_номер_фичи'_'короткое_наименование_фичи_на_русском'
2. Пользователь вручную создает файл 1_textual_description.md
3. Пользователь дает команду агенту "START feature #..." (если необходимо все реализовать без подтверждений от пользователя и не задавая ему вопросов, то для команды START указывается так же параметр --silent)
4. Агент автоматически создает файл agents/status/feature_status_'порядковый_номер_фичи'_'короткое_наименование_фичи_на_русском'.md (если он не существует) (вместо плэйсхолдеров в кавычках должны быть подставлены реальные значения без кавычек)
    В этом файле агент пишет 5 переменных:
        ## current_status=IN_PROGRESS
        ## current_run_mode=NEW
        ## last_completed_step=(присваиваем 4 если текущее значение не было заполнено или было равно нулю, оставляем без изменения если текущее значение было не равно 4 и не равно 0)
        ## last_completed_step_summary (изменяем эту переменную если текущее значение last_completed_step установили в 4, иначе оставляем без изменений)
            Создан feature_status_'порядковый_номер_фичи'_'короткое_наименование_фичи_на_русском'.md
            Начинаем реализацию фичи
        ## current_step=(присваиваем 5 если текущее значение last_completed_step установили в 4, иначе оставляем без изменений)
        ## step_changed_files
5. Далее идем с шага current_step (если current_step=5, то идем с шага 6)
6. Если текущий запуск в режиме NEW (а не REVISION) и такая фича была ранее проработана (существуют соответствующие папки/файлы относящиеся именно к этой фиче кроме 1_textual_description.md и feature_status_), то выдается соответствующее предупреждение пользователю с деталями из файла feature_status_... и продолжаем только с его согласия [подтверждение от пользователя]
7. Агент автоматически создает файл 2_user_stories.md (описывает stories для всех задействованных ролей) на основе 1_textual_description.md
8. Агент автоматически создает файл 3_acceptance_criteria.md на основе существующих файлов для этой фичи (описывает критерии приемки без выполнения которых невозможно завершить фичу и успешно пройти все тесты)
9. Агент автоматически создает файл 4_bdd_scenarios.gherkin для проверки этой фичи, на основе существующих файлов для этой фичи (сценарий всегда начинается с логина пользователя с соответствующей ролью)
10. Агент запрашивает у пользователя ревью и подтверждение созданных файлов для этой фичи (2_user_stories.md, 3_acceptance_criteria.md, 4_bdd_scenarios.gherkin) [подтверждение от пользователя]
11. Пользователь вносит необходимые изменения в созданные для этой фичи файлы (2_user_stories.md, 3_acceptance_criteria.md, 4_bdd_scenarios.gherkin) и просит агента продолжить
---
12. Агент автоматически создает необходимые макеты в формате svg в папке 5_ui_layouts (это схематичные макеты показывающие расположение элементов ввода - без стилей) (этот шаг и два следующих выполняются только если в рамках этой фичи нужно дорабатывать frontend)
13. Агент просит у пользователя подтверждение созданных макетов или комментарии по их доработке (это происходит в цикле пока пользователь не одобрит макеты) [подтверждение от пользователя]
14. Пользователь одобряет макеты или просит агента внести изменения
---
15. Агент автоматически дорабатывает agents/analytics/ba/end_to_end/end_to_end_process_bpmn.plantuml (создает файл если он не существует); диаграмма описывает сквозной продуктовый процесс платформы E-Trainer и интеграцию всех реализуемых фич в пользовательский/business flow, а не процесс разработки фич
16. Агент автоматически дорабатывает agents/analytics/ba/end_to_end/end_to_end_process_description.md (создает файл если он не существует); описание должно соответствовать продуктовому BPMN процессу платформы, а не workflow delivery
17. Агент автоматически дорабатывает agents/analytics/ba/end_to_end/end_to_end_bdd_scenarios.gherkin (создает файл если он не существует) для проверки полного продуктового цикла по зеленой ветке (соответствует BPMN диаграмме полного цикла платформы); сценарии описывают поведение платформы и роли пользователей, а не шаги создания/согласования фич
18. Агент запрашивает у пользователя ревью и подтверждение файлов (end_to_end_process_bpmn.plantuml, end_to_end_process_description.md, end_to_end_bdd_scenarios.gherkin) [подтверждение от пользователя]
19. Пользователь вносит необходимые изменения в файлы (end_to_end_process_bpmn.plantuml, end_to_end_process_description.md, end_to_end_bdd_scenarios.gherkin) и просит агента продолжить
---
20. Агент автоматически создает папку agents/analytics/sa/features/feature_'порядковый_номер_фичи'_'короткое_наименование_фичи_на_русском' (вместо плэйсхолдеров в кавычках должны быть подставлены реальные значения без кавычек)
21. Агент для каждого backend модуля затронутого разработкой этой фичи выполняет пункты с 22 по 27

24. Агент автоматически создает файл feature_module_openapi_specification.yml с описанием изменений в рамках этой фичи для текущего модуля (в итоге спецификация должна соответствовать реальному Swagger этого модуля в runtime)
25. Агент автоматически создает файл feature_module_openapi_specification_description.md с описанием изменений в рамках этой фичи для текущего модуля (эндпойнты, DTOs, валидации, STR_MNEMOs etc)
26. Агент запрашивает у пользователя ревью и подтверждение созданных файлов в рамках этой фичи для текущего модуля (feature_module_er_diagram.plantuml, feature_module_er_diagram_description.md, feature_module_openapi_specification.yml, feature_module_openapi_specification_description.md)
    [подтверждение от пользователя]
27. Пользователь вносит необходимые изменения в созданные файлы в рамках этой фичи для текущего модуля (feature_module_er_diagram.plantuml, feature_module_er_diagram_description.md, feature_module_openapi_specification.yml, feature_module_openapi_specification_description.md) и просит агента продолжить
---
28. Агент автоматически создает feature_sequence_diagram.plantuml для этой фичи (описывается все взаимодействие по этой фиче между фронтом и каждым задействованным backend модулем, а так же внешними системами если применимо)
29. Агент автоматически создает feature_sequence_diagram_description.md для этой фичи (описывается все взаимодействие по этой фиче между фронтом и каждым задействованным backend модулем, а так же внешними системами если применимо)
30. Агент запрашивает у пользователя ревью и подтверждение созданных feature_sequence_diagram.plantuml и feature_sequence_diagram_description.md для этой фичи [подтверждение от пользователя]
31. Пользователь вносит необходимые изменения в feature_sequence_diagram.plantuml и feature_sequence_diagram_description.md для этой фичи и просит агента продолжить
---
32. Агент для каждого backend модуля затронутого разработкой этой фичи выполняет пункты с 33 по 37
33. Агент автоматически создает папку agents/analytics/sa/modules/module_'название_backend_модуля' (если она не существует) (вместо плэйсхолдеров в кавычках должны быть подставлены реальные значения без кавычек)

---
38. Агент автоматически дорабатывает architecture.plantuml (создает файл если он не существует) - это архитектурная схема на которой каждый модуль бэкэнда одним блоком обозначен и один модуль для фронтэнда. И связи между всеми модулями обозначены (вызовы эндпойнтов итд)

41. Пользователь вносит необходимые изменения в файлы (architecture.plantuml, architecture_description.md и файлы для затронутых модулей) и просит агента продолжить

Агент автоматически дорабатывает EndToEndApiTest.java соответствующий cквозному BDD сценарию (создает файл если он не существует) для проверки полного цикла по зеленой ветке (этот тест вызывает другие тесты для реализованных фичей, а не повторяет их) (начинается с поднятия чистых схем для каждого модуля итд)
47. Агент автоматически дорабатывает end_to_end_ui_test.spec.ts (playwright test) соответствующий cквозному BDD сценарию (создает файл если он не существует) для проверки полного цикла по зеленой ветке (этот тест вызывает другие тесты для реализованных фичей, а не повторяет их) (тест всегда начинается с логина пользователя с соответствующей ролью)

56. Если в рамках фичи во frontend появляются новые или измененные user-facing строки, агент обязан вынести их в frontend i18n dictionaries и добавить/обновить соответствующие ключи для всех поддерживаемых языков в рамках этой же задачи; hardcoded user-facing text во frontend не допускается
57. Агент генерирует код для infrastructure (создает/изменяет/удаляет соответствующие файлы/папки) в соответствии со всеми сгенерированными к этому моменту артефактами для этой фичи (и если нужно - в соответствии с артефактами для предыдущих фич тоже)

## FEATURE WORKFLOW INSTRUCTIONS
- После выполнения каждого шага выставляем переменную current_step = номер_нового_шага_который_будет_выполняться_следующим и соответствующим образом изменяем остальные переменные в agents/status/feature_status_'порядковый_номер_фичи'_'короткое_наименование_фичи_на_русском'.md
- В названиях файлов и папок после заполнения плэйсхолдеров кавычки всегда удаляем если они остались (кавычек в названиях файлов в принципе не должно быть)
- Не запрашивай лишних подтверждений у пользователя. Всегда сразу реализуй наиболее правильный по твоему мнению вариант. Подтверждение у пользователя запрашивай только на тех шагах где явно указано [подтверждение от пользователя]
- Если при запуске команды START или RESTART указан параметр --silent, то никаких вопросов пользователю задавать нельзя и никаких подтверждений запрашивать нельзя (вместо подтверждения пишем уведомление "---auto approve, started in silent mode---"), агент в этом режиме всегда сам выбирает наиболее подходящий вариант

После завершения выполнения шага всегда пиши в чат с новой строки "------Completed step #..." (пиши как нейтральное сообщение что бы модель на него не отвечала и не учитывала его в контексте, только в чате что бы было видно)

После завершения КАЖДОГО шага workflow агент обязан немедленно обновить
`agents/status/feature_status_<...>.md`.

Правила обновления:
- `last_completed_step` = номер шага, который реально полностью завершён.
- `last_completed_step_summary` = краткое описание фактически завершённого шага и созданных/обновлённых артефактов.
- `current_step` = номер следующего шага, который должен выполняться следующим.
- `current_status` = `IN_PROGRESS`, пока feature не завершена; `COMPLETED` только после шага 62.
- Добавить в ##step_changed_files новую запись в виде:
  dd.mm.yyyy hh:mm:ss - step #{step_num} - {list of changed files in a column, except feature_status_, each file from new line, add 4 spaces before each file}
- Нельзя откладывать обновление status-файла до конца серии шагов.
- Нельзя начинать новый шаг, пока status-файл не синхронизирован с фактически завершённым предыдущим шагом.
- Если выполнение было остановлено, прервано или упало, status-файл должен отражать последний реально завершённый шаг.
- Если агент обнаружил, что артефакты созданы дальше, чем указано в status-файле, он обязан сначала исправить status-файл, затем продолжить workflow.
