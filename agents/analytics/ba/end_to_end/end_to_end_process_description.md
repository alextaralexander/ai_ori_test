# Сквозной продуктовый процесс Best Ori Gin

Документ описывает пользовательский и бизнес-поток платформы, а не процесс разработки фич.

Пользователь начинает с публичной витрины Best Ori Gin: главная, новости, FAQ, документы, benefits, каталог и карточки товаров. После выбора товаров клиент переходит в корзину и оформление заказа. Партнер проходит invite/registration, активирует статус и использует партнерские сценарии: дозаказ, бонусный кошелек, отчеты, документы по выплатам и бизнес-раздел.

Feature 016 добавляет в этот поток бизнес-раздел `/business`. Партнер-лидер видит MLM dashboard текущей кампании, структуру downline, conversion funnel, активность команды, условия апгрейда и карточку партнера по `personNumber`. Эти данные помогают партнеру управлять ростом структуры до оформления заказов downline, начисления бонусов и формирования отчетов.

Feature 017 расширяет продуктовый путь офлайн-продажами через партнера. Партнер-продавец или партнер-лидер открывает `/vip-orders` или `/business/tools/order-management/vip-orders/partner-orders`, фильтрует заказы клиентов по кампании, статусам, customerId и partnerPersonNumber, а затем открывает карточку `/business/tools/order-management/vip-orders/partner-orders/:orderId`. В карточке он видит состав заказа, оплату, доставку, начисление бонусов, businessVolume, timeline событий и доступные сервисные действия. Сотрудник поддержки заказов может выполнить разрешенную сервисную корректировку, а партнер может инициировать повторный заказ клиента.

Feature 018 добавляет в этот путь операционный контур партнерского офиса. После появления офлайн-заказов офис открывает `/partner-office/all-orders`, `/partner-office/supply`, `/partner-office/supply/:supplyId`, `/partner-office/supply/orders/:orderId` и `/partner-office/report`, чтобы контролировать все заказы офиса, supply-поставки, приемку, складские движения, комплектность, недостачи, повреждения и SLA. Логистический оператор сопровождает lifecycle поставки от планирования до приемки, партнер-офис фиксирует отклонения приемки и связывает их с claim workflow, а региональный менеджер контролирует проблемные маршруты, эскалации и показатели офиса. Этот поток связывает order, WMS/1C, delivery, pickup point и claim процессы без описания механики разработки feature.

После заказов backend связывает order, bonus wallet, partner reporting и MLM-structure: объемы и квалификационные условия обновляют dashboard, отчеты и документы. WMS/1C передает остатки, сборочный конвейер обновляет статусы комплектования, доставка передает статусы и точки выдачи. При претензиях сервис учитывает компенсации и влияние на партнерские показатели.

Администратор управляет каталогами, кампаниями, PIM, заказами, офлайн-заказами клиентов партнера, претензиями, бонусной системой, compensation plan, пользователями, ролями и интеграциями. Для predefined пользовательских сообщений backend передает только `STR_MNEMO_*`, frontend локализует их через i18n dictionaries.

Технологический baseline для feature #18 фиксируется без введения новых технологий: Java 26.0.1 доступен на дату старта, но текущий Maven baseline и уже реализованные monolith modules закреплены на Java 25 и `maven.compiler.release=25`; upgrade Java 26 должен быть отдельной совместимостью задачей, чтобы не смешивать feature delivery и platform upgrade. Backend использует Spring Boot 4.0.6, Maven и XML Liquibase. Frontend web-shell использует React 19.2, TypeScript 6.0, Ant Design и Vite через latest-зависимости package.json на дату старта задачи 27.04.2026.
