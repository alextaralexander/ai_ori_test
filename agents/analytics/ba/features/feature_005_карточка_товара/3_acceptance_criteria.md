# Acceptance criteria. Feature 005. Карточка товара

## AC-005-01. Маршрут и загрузка карточки
- Страница доступна по маршруту `/product/:productCode`.
- При открытии существующего опубликованного товара отображается детальная карточка без необходимости авторизации.
- При неизвестном, неопубликованном или недоступном для публичного просмотра `productCode` пользователь видит локализованное состояние "товар не найден" без технической ошибки.
- Прямая ссылка на карточку сохраняет выбранный продукт и не требует предварительного поиска в каталоге.

## AC-005-02. Основная информация
- Карточка показывает название, код товара, категорию, бренд, объем или размер, главное изображение и список дополнительных изображений, если они есть.
- Если дополнительные изображения отсутствуют, галерея не показывает пустые элементы и не ломает компоновку.
- Публичные промо-метки и campaign-метки отображаются только для опубликованных и разрешенных к показу значений.
- Данные карточки согласованы с выдачей каталога feature #4 по названию, коду, цене, промо-меткам и статусу доступности.

## AC-005-03. Цена, промо и доступность
- Гость видит публичную цену, промо-цену при наличии и общий статус доступности.
- Авторизованный покупатель и партнер видят условия с учетом текущей кампании, пользовательского статуса и партнерского контекста.
- Статусы доступности различают минимум: доступен, мало осталось, недоступен, скоро появится, снят с продажи.
- Ограничения заказа включают минимальное количество, максимальное количество на строку и доступный остаток, если эти данные переданы backend.

## AC-005-04. Информационные блоки
- Описание, способ применения, характеристики, состав и вложения отображаются в отдельных логических блоках или вкладках.
- Пустые необязательные блоки скрываются, но обязательная основная информация карточки остается видимой.
- Вложения открываются или скачиваются по валидным ссылкам, если они переданы backend.
- Состав и характеристики отображаются как данные продукта, а не как hardcoded текст frontend.

## AC-005-05. Рекомендации
- Карточка показывает связанные товары, cross-sell и related offers, если backend вернул соответствующие элементы.
- Рекомендации не включают текущий товар как отдельную карточку рекомендации.
- Если рекомендаций нет, блок рекомендаций скрывается без пустого состояния, мешающего покупке.
- Карточки рекомендаций позволяют перейти на соответствующую страницу `/product/:productCode`.

## AC-005-06. Добавление в корзину
- Авторизованный покупатель или партнер может выбрать количество в допустимых пределах и добавить доступный товар в корзину.
- После успешного добавления отображается локализованное подтверждение, обновляется краткая информация о корзине и остается возможность перейти к checkout.
- Гость при попытке покупки получает локализованное предложение войти или зарегистрироваться без потери контекста товара.
- Недоступный товар, превышение лимита или некорректное количество не добавляются в корзину; пользователь видит локализованную причину.

## AC-005-07. Быстрый checkout
- Кнопка быстрого перехода к checkout активна только после успешного добавления товара или при уже существующей валидной строке корзины для этого товара.
- Переход в checkout сохраняет выбранный товар, количество и партнерский контекст.
- Если checkout еще не реализован как отдельная фича, frontend использует стабильный route contract `/checkout` и не показывает техническую ошибку при переходе.

## AC-005-08. Backend contract
- Backend endpoint карточки возвращает структурированные данные продукта, медиа, цены, доступности, ограничений, информационных блоков, вложений и рекомендаций.
- Backend не передает hardcoded user-facing тексты для предопределенных сообщений; для ошибок и статусов, требующих UI-текста, используются mnemonic-коды `STR_MNEMO_*`.
- DTO не содержит лишних внутренних полей администрирования, черновиков и технических признаков, не предназначенных для frontend.
- Swagger/OpenAPI для модуля catalog должен отражать endpoint карточки, DTO, варианты ошибок и endpoint добавления в корзину.

## AC-005-09. Frontend i18n
- Все новые user-facing строки карточки, ошибок, кнопок, статусов, вкладок, подсказок и aria-label вынесены в i18n-словари.
- Новые ключи добавлены для всех поддерживаемых языков frontend.
- React-компоненты не содержат новых hardcoded пользовательских строк.

## AC-005-10. Доступность и адаптивность
- Галерея, выбор количества, вкладки/секции информации, кнопка добавления в корзину и переход к checkout доступны с клавиатуры.
- Интерактивные элементы имеют понятные accessible names.
- Desktop-макет показывает медиа и покупательский блок рядом, а информационные секции и рекомендации ниже.
- Mobile-макет показывает покупательский блок до длинных информационных секций, чтобы покупка была доступна без чрезмерной прокрутки.

## AC-005-11. Ошибки и пустые состояния
- Ошибка загрузки карточки, отсутствие товара, недоступность товара и ошибка добавления в корзину имеют отдельные локализованные состояния.
- Повторная попытка загрузки или добавления не создает дублирующие строки корзины при одинаковом товаре и контексте.
- Frontend не показывает stack trace, raw exception или технический JSON пользователю.

## AC-005-12. Тестовое покрытие
- Managed API test начинается с логина подходящей роли и проверяет успешное получение карточки, недоступный товар и добавление доступного товара в корзину.
- Managed UI test начинается с логина подходящей роли и проверяет открытие карточки, галерею/информационные блоки, добавление в корзину и переход к checkout.
- End-to-end managed tests агрегируют реальный тест feature #5 вместе с предыдущими feature-тестами зеленого пути, а не содержат placeholder assertions.
- Runtime-копии managed tests синхронизированы из `agents/tests/` по `agents/tests/targets.yml` и содержат marker comment.

## Версионная база
Новые технологии не вводятся. Используются текущие версии существующего Spring Boot monolith, Java/Maven baseline, React/TypeScript web-shell и Ant Design-compatible UI-подход проекта. При реализации нельзя добавлять альтернативный frontend state manager или отдельный catalog service без отдельного архитектурного решения.
