# Acceptance criteria. Feature 006. Цифровые каталоги и PDF-материалы

## AC-006-01. Маршруты цифровых каталогов
- Страница текущего каталога доступна по маршруту `/products/digital-catalogue-current`.
- Страница следующего каталога доступна по маршруту `/products/digital-catalogue-next`.
- Оба маршрута открываются без авторизации, но содержимое следующего каталога может быть ограничено по роли и статусу публикации.
- При отсутствии опубликованного выпуска пользователь видит локализованное пустое состояние без stack trace, raw JSON или технического сообщения.

## AC-006-02. Данные выпуска каталога
- Для каждого отображаемого выпуска показаны название, тип периода, дата начала, дата окончания и статус публикации.
- Текущий каталог выбирается по активному периоду кампании и статусу `PUBLISHED`.
- Следующий каталог выбирается по ближайшему будущему периоду и статусу, разрешенному для соответствующей роли.
- Если найдено несколько подходящих выпусков одного типа, frontend получает детерминированный порядок, а backend не возвращает конфликтующие активные выпуски как равнозначные.

## AC-006-03. Навигация по страницам
- Пользователь может перейти к следующей и предыдущей странице каталога.
- Пользователь видит текущий номер страницы и общее количество страниц.
- Пользователь может открыть конкретную страницу по номеру или миниатюре, если миниатюры доступны.
- Недоступные или отсутствующие изображения страниц не ломают viewer и показывают контролируемое состояние.

## AC-006-04. Встроенный PDF viewer
- PDF-материал открывается внутри страницы каталога без обязательного скачивания.
- Viewer поддерживает минимум: увеличение, уменьшение, сброс масштаба, скачивание и создание ссылки для шаринга, если действие разрешено правами материала.
- Кнопки viewer имеют локализованные labels и accessible names.
- Если PDF временно недоступен, пользователь видит локализованное состояние ошибки и возможность повторить загрузку.

## AC-006-05. Связанные PDF-материалы и брошюры
- Выпуск может содержать несколько материалов: основной каталог, брошюры, промо-листы и дополнительные документы.
- Материалы отображаются только если они опубликованы и разрешены текущей роли.
- Для каждого материала показываются тип, название, размер файла при наличии и доступные действия.
- Скачивание и шаринг не раскрывают приватные материалы пользователям без соответствующих прав.

## AC-006-06. Переходы в карточки товаров
- Страница каталога может содержать интерактивные зоны, связанные с `productCode`.
- При выборе зоны пользователь переходит на `/product/:productCode`.
- Переход сохраняет campaign-контекст выпуска, чтобы карточка товара использовала согласованные цену, доступность и промо-условия.
- Если товар скрыт, снят с публикации или недоступен текущей роли, пользователь видит контролируемое состояние карточки товара по правилам feature #5.

## AC-006-07. Ролевая видимость
- Гость видит только публично опубликованные материалы текущего каталога и публично разрешенный следующий каталог.
- Авторизованный пользователь видит материалы, разрешенные его роли и статусу.
- Контент-менеджер и каталожный менеджер могут видеть preview-материалы только в пределах разрешенного режима предпросмотра, без раскрытия preview-ссылок публичным пользователям.
- Backend не возвращает материалы, интерактивные зоны и служебные признаки, которые не должны быть доступны текущему пользователю.

## AC-006-08. Статусы публикации
- Поддержаны минимум статусы выпуска: черновик, запланирован, опубликован, архивирован.
- Публичные маршруты не показывают черновики и архивные выпуски как активные каталоги.
- Следующий каталог не отображается раньше разрешенного времени предпросмотра для роли.
- Дата действия выпуска отображается в едином формате через frontend i18n/locale formatting.

## AC-006-09. Backend contract
- Backend endpoint списка/деталей цифрового каталога возвращает структурированные данные выпуска, страниц, материалов, интерактивных зон, прав действий и статусов.
- Backend endpoint получения PDF/материала проверяет права доступа перед выдачей URL или контента.
- Backend не передает hardcoded user-facing тексты для предопределенных сообщений; для ошибок и контролируемых состояний используются mnemonic-коды `STR_MNEMO_*`.
- Swagger/OpenAPI для owning backend module отражает endpoints каталога, DTO, варианты ошибок доступа и контракты материалов.

## AC-006-10. Frontend i18n
- Все новые user-facing строки цифровых каталогов, кнопок viewer, статусов, ошибок, пустых состояний и aria-label вынесены в frontend i18n dictionaries.
- Новые ключи добавлены для всех поддерживаемых языков frontend.
- React-компоненты не содержат новых hardcoded пользовательских строк.
- Backend mnemonic-коды, которые могут попасть во frontend, добавлены во все поддерживаемые frontend dictionaries.

## AC-006-11. Адаптивность и доступность
- Desktop layout показывает область viewer, навигацию по страницам и список материалов без перекрытий.
- Mobile layout сохраняет доступ к листанию, масштабу, скачиванию и переходу к товару без горизонтальной прокрутки страницы.
- Все интерактивные элементы доступны с клавиатуры и имеют видимый focus state.
- Viewer не блокирует базовую навигацию браузера и не создает ловушку фокуса вне модальных состояний.

## AC-006-12. Производительность и устойчивость
- Первый экран страницы каталога показывает метаданные выпуска и состояние загрузки viewer без ожидания полной загрузки всех PDF-страниц.
- Тяжелые страницы, миниатюры и PDF загружаются лениво или по требованию.
- Повторные запросы к одному материалу не создают неконтролируемого дублирования загрузок.
- При сетевой ошибке пользователь может повторить загрузку материала или вернуться к списку материалов.

## AC-006-13. Тестовое покрытие
- Managed API test начинается с логина подходящей роли и проверяет получение текущего каталога, следующего каталога, запрет на приватный материал и наличие интерактивной зоны с productCode.
- Managed UI test начинается с логина подходящей роли и проверяет открытие текущего каталога, навигацию по страницам, zoom/download/share viewer и переход в карточку товара.
- End-to-end managed tests агрегируют реальный тест feature #6 вместе с предыдущими feature-тестами зеленого пути, а не содержат placeholder assertions.
- Runtime-копии managed tests синхронизированы из `agents/tests/` по `agents/tests/targets.yml` и содержат marker comment.

## Версионная база
Новые технологии не вводятся. Используются текущие версии существующего Spring Boot monolith, Java/Maven baseline, React/TypeScript web-shell и Ant Design-compatible UI-подход проекта. При реализации нельзя добавлять отдельный PDF service, frontend state manager или альтернативный viewer stack без отдельного архитектурного решения и зафиксированной совместимости.
