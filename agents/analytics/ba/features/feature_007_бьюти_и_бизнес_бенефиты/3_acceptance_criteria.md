# Acceptance criteria. Feature 007. Бьюти- и бизнес-бенефиты

## AC-007-01. Публичные маршруты лендингов
- Маршрут `/beauty-benefits` открывает публичный beauty landing без обязательной авторизации.
- Маршрут `/business-benefits` открывает публичный business landing без обязательной авторизации.
- Маршруты `/member-benefits`, `/vip-customer-benefits` и `/the-new-oriflame-app` открывают соответствующие страницы выгод и мобильного приложения без обязательной авторизации.
- При ошибке загрузки данных пользователь видит локализованное контролируемое состояние без raw JSON, stack trace или технического сообщения.

## AC-007-02. Персонализированные referral-маршруты
- Маршрут `/beauty-benefits/:code` принимает referral/invite-код из URL и сохраняет его в landing-context.
- Маршрут `/business-benefits/:code` принимает referral/invite-код из URL и сохраняет его в landing-context.
- Код передается в CTA регистрации, связи со спонсором и аналитические события без ручного повторного ввода пользователем.
- Код нормализуется безопасно: пробелы и недопустимые символы не приводят к XSS, сломанному URL или падению страницы.

## AC-007-03. Валидация referral-кода
- Если код найден и активен, страница отображает персонализированный контекст спонсора только в объеме, разрешенном публичным контрактом.
- Если код не найден, истек или отключен, пользователь видит локализованное состояние и может продолжить без персонализации либо перейти к вводу другого кода в регистрации.
- Backend не возвращает лишние персональные данные спонсора через публичный endpoint.
- Ошибки referral-кода передаются во frontend как mnemonic-коды `STR_MNEMO_*` или структурированные статусы без hardcoded user-facing текста.

## AC-007-04. CTA и переходы
- На каждом лендинге есть релевантные CTA: регистрация, активация выгод, просмотр каталога, установка приложения и контакт со спонсором, если действие применимо к странице.
- Переход в регистрацию сохраняет `landingType`, `variant`, `code` и `campaignId`, если эти значения доступны.
- Переход к каталогу ведет на существующие маршруты цифровых каталогов или товарных сценариев без потери campaign-контекста.
- Недоступный CTA не отображается как активная кнопка и не ведет на пустую или техническую страницу.

## AC-007-05. Контент beauty landing
- Beauty landing объясняет преимущества продукции, каталога, welcome benefits, cashback, персональных предложений и простого старта как покупателя.
- Первый экран содержит основной value proposition и главный CTA.
- Вторичные блоки раскрывают выгоды без необходимости регистрации.
- Контент не содержит обещаний, которые невозможно подтвердить правилами кампании или будущими договорами.

## AC-007-06. Контент business landing
- Business landing объясняет партнерскую программу, community-модель, offline sales, бонусы, referral benefits и шаги старта.
- Страница явно отделяет потенциальные возможности от гарантированного дохода и содержит корректный дисклеймер.
- CTA регистрации партнера сохраняет referral-контекст.
- Персонализированная версия с кодом дает возможность перейти к контакту со спонсором, если спонсор доступен.

## AC-007-07. Member и VIP customer benefits
- `/member-benefits` показывает преимущества участника, персональные предложения, welcome benefits и связь с каталогом.
- `/vip-customer-benefits` показывает VIP-выгоды, cashback, специальные цены и условия перехода к регистрации.
- Страницы не требуют партнерского статуса для просмотра публичной информации.
- Все условия выгод отображаются как маркетинговые информационные блоки, а не как подтвержденные начисления на аккаунт до регистрации.

## AC-007-08. Страница мобильного приложения
- `/the-new-oriflame-app` описывает возможности приложения: каталог, покупки, уведомления, личные выгоды и партнерское взаимодействие.
- CTA установки приложения ведет на настроенную ссылку или показывает локализованное состояние, если ссылка временно недоступна.
- Событие клика по установке приложения фиксируется с landingType `APP`.
- Отсутствие store-link не ломает остальные CTA страницы.

## AC-007-09. A/B-ready варианты
- Контракт лендинга поддерживает `variant` для выбора набора блоков и офферов.
- Если variant не задан или неизвестен, используется стабильный default-вариант.
- Набор блоков возвращается в детерминированном порядке с типом блока, ключом контента, CTA и display conditions.
- Frontend не содержит hardcoded ветвления по экспериментальным текстам; user-facing тексты берутся из i18n или управляемого контента.

## AC-007-10. Аналитика конверсии
- При просмотре лендинга отправляется событие view с `landingType`, `variant`, `code`, `campaignId`, route и timestamp.
- При клике CTA отправляется событие click с типом CTA и тем же контекстом.
- Переход в регистрацию, установка приложения и контакт со спонсором фиксируются отдельными событиями.
- Недоступность аналитического endpoint не блокирует рендер страницы, клики CTA и переходы пользователя.

## AC-007-11. Backend contract
- Backend предоставляет endpoint получения landing payload по типу лендинга, optional referral-коду, campaignId и variant.
- Backend предоставляет endpoint регистрации conversion event или принимает его через существующий analytics contract, если такой contract уже есть.
- Backend contract содержит только структурированные данные, идентификаторы блоков, CTA-типы, статусы и mnemonic-коды для предопределенных сообщений.
- Swagger/OpenAPI owning module отражает новые endpoints, DTO, ошибки referral-кода, conversion payload и ограничения данных спонсора.

## AC-007-12. Frontend i18n
- Все новые user-facing строки лендингов, CTA, ошибок referral-кода, дисклеймеров, пустых состояний, SEO titles и aria-label вынесены в frontend i18n dictionaries.
- Новые ключи добавлены во все поддерживаемые языки frontend в рамках этой же задачи.
- React-компоненты не содержат новых hardcoded пользовательских строк.
- Backend mnemonic-коды, которые могут попасть во frontend, добавлены во все поддерживаемые frontend dictionaries.

## AC-007-13. SEO, metadata и шаринг
- Каждая страница имеет корректные title, description и canonical metadata.
- Персонализированные URL с кодом не раскрывают персональные данные в metadata сверх публично разрешенного объема.
- Open Graph/Twitter preview-данные соответствуют типу страницы.
- Неверный referral-код не должен приводить к индексированию технического состояния ошибки.

## AC-007-14. Адаптивность и доступность
- Desktop layout показывает первый экран, блоки выгод, CTA, дисклеймеры и secondary links без перекрытий.
- Mobile layout сохраняет читаемость блоков и CTA без горизонтального скролла.
- Все CTA доступны с клавиатуры, имеют видимый focus state и понятные accessible names.
- Динамические состояния загрузки, ошибки и персонализации объявляются доступным способом для assistive technologies.

## AC-007-15. Производительность и устойчивость
- Первый экран и основной CTA отображаются до загрузки вторичных медиа и аналитики.
- Медиа и вторичные блоки загружаются лениво или не блокируют critical path.
- Повторный переход между лендингами не теряет referral-контекст в рамках пользовательской сессии.
- При сетевой ошибке payload пользователь может повторить загрузку или перейти к базовой регистрации без персонализации.

## AC-007-16. Безопасность
- Referral-код обрабатывается как untrusted input и не вставляется в DOM без безопасного escaping.
- Публичный sponsor payload не содержит email, телефона или внутренних идентификаторов, если они не разрешены публичным контрактом.
- Conversion events не принимают произвольные user-facing тексты от клиента.
- Frontend не доверяет route-параметрам при формировании ссылок регистрации и использует безопасный URL builder.

## AC-007-17. Тестовое покрытие
- Managed API test начинается с логина подходящей роли или явно фиксирует публичный guest-контекст, затем проверяет payload для beauty, business, member, VIP и app landing.
- Managed API test проверяет активный referral-код, неизвестный referral-код и сохранение контекста в conversion event.
- Managed UI test начинается с входа подходящей роли или публичного открытия страницы, проверяет основные маршруты, CTA, referral URL, мобильную ширину и локализованное состояние неверного кода.
- End-to-end managed tests агрегируют реальный тест feature #7 вместе с предыдущими feature-тестами зеленого пути, а не содержат заглушечные assertions.
- Runtime-копии managed tests синхронизированы из `agents/tests/` по `agents/tests/targets.yml` и содержат marker comment.

## Версионная база
Новые технологии не вводятся. Используются текущие версии существующего Spring Boot monolith, Java/Maven baseline, React/TypeScript web-shell и Ant Design-compatible UI-подход проекта. При реализации нельзя добавлять отдельный CMS, frontend state manager или внешний A/B testing SDK без отдельного архитектурного решения, зафиксированной совместимости и обновления соответствующих workflow-артефактов.
