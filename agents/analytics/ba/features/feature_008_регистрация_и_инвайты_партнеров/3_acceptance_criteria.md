# Acceptance criteria. Feature 008. Регистрация и инвайты партнеров

## AC-008-01. Публичные маршруты регистрации
- Маршрут `/invite/beauty-partner-registration` открывает публичную форму регистрации beauty partner без обязательной авторизации.
- Маршрут `/invite/business-partner-registration` открывает публичную форму регистрации business partner без обязательной авторизации.
- Оба маршрута принимают optional invite/referral-код из URL, query-параметра или сохраненного landing-context.
- При ошибке загрузки конфигурации регистрации пользователь видит локализованное контролируемое состояние без raw JSON, stack trace или технического текста.

## AC-008-02. Сохранение marketing и referral context
- Регистрация принимает `landingType`, `campaignId`, `variant`, `sourceRoute` и referral/invite-код от benefit landing pages.
- Context сохраняется при переходах между шагами формы, перезагрузке страницы и возврате после подтверждения контакта.
- Если context из URL конфликтует с context из сессии, система применяет детерминированное правило приоритета и фиксирует конфликт в audit/event stream.
- Недопустимые символы в коде не приводят к XSS, открытому redirect, падению формы или повреждению URL.

## AC-008-03. Валидация invite/referral-кода
- Активный код связывает заявку с публично разрешенным sponsor context и выбранным onboarding type.
- Неизвестный, истекший, отключенный, уже использованный или несовместимый с типом регистрации код возвращает структурированный статус и mnemonic-код `STR_MNEMO_*`.
- Backend не возвращает во frontend hardcoded user-facing текст ошибки.
- Публичный sponsor payload не содержит email, телефона, внутренних user id, финансовых показателей или иных закрытых данных.

## AC-008-04. Пошаговая форма регистрации
- Форма собирает минимально необходимые персональные данные, контактный канал, выбранный сценарий beauty/business, согласия и invite context.
- Каждый шаг валидируется до перехода дальше, ошибки показываются рядом с полями и доступны для assistive technologies.
- Пользователь может вернуться к предыдущему шагу без потери уже введенных данных.
- Повторная отправка формы с тем же idempotency key не создает дубли заявки, партнера или sponsor attribution.

## AC-008-05. Согласия и правовые условия
- Регистрация требует подтверждения актуальных обязательных согласий: правила участия, обработка персональных данных и применимые коммерческие условия.
- Версия каждого согласия сохраняется вместе с заявкой.
- Без обязательных согласий заявка не отправляется.
- Тексты согласий, labels и ошибки отображаются через frontend i18n или управляемый юридический контент, а не через hardcoded backend text.

## AC-008-06. Создание регистрационной заявки
- После успешной отправки создается registration application со статусом `PENDING_CONTACT_CONFIRMATION`, `PENDING_CRM_REVIEW` или `READY_FOR_ACTIVATION` согласно бизнес-правилам.
- Заявка содержит source, campaignId, invite/referral code, sponsor attribution, onboarding type и audit metadata.
- Дубликат по email/телефону/идентификатору возвращает контролируемый статус и не создает второго активного кандидата.
- Пользователь видит следующий шаг: подтвердить контакт, дождаться проверки, перейти к активации или войти в существующий аккаунт.

## AC-008-07. Интеграция с CRM/маркетинговыми системами
- Событие лида отправляется в CRM/маркетинговый контур с разрешенными полями: applicationId, source, campaignId, landingType, onboardingType, sponsor code и статус.
- Временная недоступность CRM не блокирует сохранение заявки, если локальная транзакция успешна.
- Ошибка CRM фиксируется в retry/audit контуре без показа пользователю технических деталей.
- Повторная отправка CRM-события идемпотентна по applicationId/eventId.

## AC-008-08. Маршрут активации партнера
- Маршрут `/invite/partners-activation` доступен приглашенному партнеру и открывает состояние активации по application/activation token.
- Token валидируется по сроку действия, статусу заявки, привязке к контакту и одноразовости критичных операций.
- Истекший, неверный или уже использованный token показывает локализованное контролируемое состояние.
- Активация не раскрывает данные другого кандидата при подмене token или application id.

## AC-008-09. Подтверждение контакта
- Приглашенный партнер может подтвердить email или телефон одноразовым кодом согласно настроенному каналу.
- Повторная отправка кода ограничена rate limit и не раскрывает, зарегистрирован ли чужой контакт.
- Неверный или истекший код возвращает mnemonic-код `STR_MNEMO_*` и не активирует партнера.
- Успешное подтверждение переводит заявку к следующему разрешенному статусу.

## AC-008-10. Активация партнерского статуса
- При выполнении всех условий система активирует partner profile и связывает его со sponsor attribution.
- Новый партнер получает начальный партнерский статус, доступные роли/permissions и базовую связь с будущей MLM-структурой.
- Фича не рассчитывает MLM-бонусы, комиссии и квалификации; она только создает корректную входную связь для последующих фич.
- Повторный запрос активации по уже активированной заявке возвращает текущий результат без дублей.

## AC-008-11. Personal referral link
- После активации партнер получает personal referral code и personal referral link для beauty/business приглашений.
- Код уникален, не раскрывает внутренний id пользователя и может быть безопасно размещен в публичных ссылках.
- Ссылка ведет в существующие benefit или invite маршруты с сохранением attribution.
- Если генерация ссылки временно недоступна, партнер видит локализованное состояние и может повторить действие позже.

## AC-008-12. Sponsor cabinet
- Маршрут `/invite/sponsor-cabinet` доступен авторизованному спонсору/партнеру.
- Sponsor cabinet показывает список invite-записей и кандидатов только текущего спонсора.
- Для каждого invite отображаются разрешенные поля: тип, статус, дата создания, срок действия, публичный прогресс и безопасная краткая информация о кандидате.
- Спонсор не может увидеть приглашения другого партнера через подмену id, query-параметров или route state.

## AC-008-13. Создание и повторная отправка invite
- Спонсор может создать invite для beauty или business сценария, если его статус и текущая кампания позволяют это действие.
- Созданный invite получает срок действия, status, target onboarding type, source и уникальный безопасный code.
- Спонсор может скопировать ссылку или инициировать повторную отправку через поддерживаемый канал без создания дубля, если invite еще активен.
- Недоступное действие отображается как контролируемое состояние и не ведет на пустую страницу.

## AC-008-14. Статусы приглашений
- Статусы invite и registration application синхронно отражают ключевые этапы: created, opened, registration started, submitted, contact pending, active, expired, rejected.
- Истечение срока invite обрабатывается плановой задачей или проверкой при чтении без ручного вмешательства пользователя.
- Sponsor cabinet не показывает устаревший статус после успешной активации кандидата.
- Статусные тексты во frontend берутся из i18n dictionaries.

## AC-008-15. Analytics и audit
- Система фиксирует события открытия invite, начала регистрации, отправки формы, ошибки кода, подтверждения контакта, активации и создания referral link.
- События содержат eventId, timestamp, source, campaignId, onboardingType и безопасный sponsor/referral identifier.
- Audit trail фиксирует создание invite, изменение статуса, конфликт атрибуции и активацию без записи паролей, одноразовых кодов или лишних персональных данных.
- Недоступность аналитики не блокирует регистрацию, активацию и sponsor cabinet.

## AC-008-16. Frontend i18n
- Все новые user-facing строки регистрации, активации, sponsor cabinet, CTA, form labels, validation messages, статусы и SEO metadata вынесены в frontend i18n dictionaries.
- Новые ключи добавлены во все поддерживаемые frontend languages в рамках этой же задачи.
- React-компоненты не содержат новых hardcoded пользовательских строк.
- Backend mnemonic-коды, которые могут попасть во frontend, добавлены во все поддерживаемые frontend dictionaries.

## AC-008-17. Backend contract
- Backend предоставляет endpoint валидации invite/referral-кода.
- Backend предоставляет endpoint создания registration application.
- Backend предоставляет endpoint чтения и выполнения activation flow.
- Backend предоставляет endpoint sponsor cabinet для списка invite, создания invite и повторной отправки/копирования invite link.
- Все endpoints отражены в OpenAPI owning module и используют DTO без hardcoded user-facing текста.

## AC-008-18. Безопасность
- Все публичные token/code параметры обрабатываются как untrusted input.
- Для form submit, activation и sponsor cabinet применяются CSRF/session/JWT правила, соответствующие текущей security-модели проекта.
- Rate limit применяется к валидации invite-кода, отправке формы, отправке одноразового кода и повторной отправке invite.
- Логи и ошибки не содержат паролей, одноразовых кодов, полных token, лишних персональных данных и stack trace для пользователя.

## AC-008-19. Адаптивность и доступность
- Desktop layout регистрации, активации и sponsor cabinet не содержит перекрытий, горизонтального скролла и нечитабельных блоков.
- Mobile layout позволяет пройти регистрацию и активацию одной колонкой без потери CTA и статусов.
- Поля формы имеют labels, ошибки связаны с полями, focus state видим, шаги формы объявляются доступным способом.
- Sponsor cabinet поддерживает keyboard navigation по таблице/списку и действиям invite.

## AC-008-20. Тестовое покрытие
- Managed API test начинается с логина подходящей роли или явно фиксирует публичный guest-контекст, затем проверяет регистрацию beauty/business, active/invalid/expired invite-коды, создание заявки, активацию и sponsor cabinet.
- Managed API test проверяет идемпотентность отправки формы и отсутствие дублей при повторном submit.
- Managed UI test начинается с входа подходящей роли или публичного открытия страницы, проверяет маршруты регистрации, invite context, ошибки кода, активацию, sponsor cabinet и мобильную ширину.
- Managed end-to-end tests агрегируют реальный тест feature #8 вместе с предыдущими feature-тестами зеленого пути, а не содержат заглушечные assertions.
- Runtime-копии managed tests синхронизированы из `agents/tests/` по `agents/tests/targets.yml` и содержат marker comment.

## Версионная база
Новые технологии не вводятся. Используются текущие версии существующего Spring Boot monolith, Java/Maven baseline, React/TypeScript web-shell и Ant Design-compatible UI-подход проекта. При реализации нельзя добавлять отдельный identity provider, CRM SDK, notification broker или внешний invite-service без отдельного архитектурного решения, документированной совместимости и обновления workflow-артефактов.
