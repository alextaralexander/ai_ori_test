# Acceptance criteria. Feature 009. Корзина и промо-предложения

## AC-009-01. Маршруты корзины
- Маршрут `/cart` доступен авторизованному покупателю и партнеру и открывает основную корзину текущего пользователя.
- Маршрут `/cart/shopping-offers` показывает промо-предложения, применимые к основной корзине.
- Маршрут `/cart/supplementary` доступен партнеру и открывает отдельную корзину дозаказа без смешивания с основной корзиной.
- Маршрут `/cart/supplementary/shopping-offers` показывает промо-предложения, применимые только к supplementary order.
- Неавторизованный пользователь при открытии защищенных маршрутов получает локализованный сценарий входа с сохранением intended route.

## AC-009-02. Состав корзины
- Корзина принимает товары, добавленные из feature #4 и feature #5, с сохранением `productCode`, количества, campaign context и пользовательского/партнерского context.
- Одна и та же позиция в одинаковом контексте не дублируется, а увеличивает количество в существующей строке с учетом лимитов.
- Пользователь может изменить количество позиции в допустимых пределах и удалить позицию.
- Пустая корзина показывает локализованное состояние и ссылку возврата в каталог без технических деталей.

## AC-009-03. Пересчет цен и итогов
- После каждого добавления, удаления или изменения количества backend возвращает актуальные строки корзины, subtotal, скидки, промо-выгоды, ожидаемые подарки, shipping threshold progress и grand total.
- Пересчет учитывает текущую кампанию каталога, role segment, partner context, доступные остатки и активные промо-правила.
- Frontend отображает состояние пересчета без потери уже показанных строк и без двойного применения скидок.
- Повторный запрос пересчета с тем же idempotency key не создает новые строки и не меняет итог сверх ожидаемого пересчета.

## AC-009-04. Доступность, резервы и ограничения
- Каждая строка корзины имеет структурированный availability status: `AVAILABLE`, `LOW_STOCK`, `RESERVED`, `PARTIALLY_AVAILABLE`, `UNAVAILABLE` или `REMOVED_FROM_CAMPAIGN`.
- Если запрошенное количество превышает доступный остаток или лимит, система возвращает допустимое количество и mnemonic-код причины.
- Товар, закрытый для роли, кампании или канала заказа, нельзя провести к checkout.
- Backend не возвращает во frontend hardcoded user-facing текст причин; используются данные и `STR_MNEMO_*` для предопределенных сообщений.

## AC-009-05. Промо-предложения основной корзины
- `/cart/shopping-offers` возвращает предложения текущей кампании: наборы, подарки, cross-sell, upsell, retention offers и предложения бесплатной доставки по специальному продукту.
- Предложение содержит тип, применимость, required/remaining condition, связанные товары, ожидаемую выгоду и structured status.
- Недействующие, истекшие, несовместимые с ролью или недоступные по остаткам предложения не показываются как доступные.
- Добавление промо-предложения обновляет корзину и итоги тем же контрактом, что и обычное добавление товара.

## AC-009-06. Supplementary order
- Supplementary cart хранится отдельно от основной корзины и имеет собственный type/context.
- Партнер может добавлять, удалять и менять количество товаров в supplementary cart в рамках правил дозаказа.
- Основная корзина и supplementary cart не используют общие строки, итоги и промо-состояния.
- Если партнеру недоступен дозаказ, `/cart/supplementary` и `/cart/supplementary/shopping-offers` показывают контролируемое локализованное состояние.

## AC-009-07. Промо для supplementary order
- `/cart/supplementary/shopping-offers` показывает только предложения, разрешенные для дозаказа, partner role segment и текущей кампании.
- Промо supplementary order не может примениться к основной корзине через подмену route, cartId или request body.
- Добавление supplementary offer пересчитывает только supplementary cart.
- Офферы, завязанные на основную корзину, не появляются в списке supplementary shopping offers.

## AC-009-08. Подготовка к checkout
- Переход к checkout доступен только для непустой валидной корзины без blocking availability statuses и нарушенных бизнес-ограничений.
- Перед переходом выполняется server-side validation состава, цен, доступности, campaign status, role permissions и benefit rules.
- Если checkout еще не реализован как полноценная фича, frontend использует стабильный route contract `/checkout` и показывает контролируемое состояние при недоступности следующего шага.
- Невалидная корзина возвращает список строк и причин, которые нужно исправить до checkout.

## AC-009-09. Поддержка заказов
- Авторизованный сотрудник поддержки заказов с разрешенными правами может получить read-only или controlled support view корзины пользователя.
- Сотрудник поддержки видит структурированные причины блокировки и применимости промо без доступа к закрытым финансовым данным сверх роли.
- Любое изменение корзины от имени пользователя требует audit metadata и не может обходить складские, каталожные, role-based и benefit ограничения.
- Пользовательские данные корзины недоступны другим покупателям и партнерам через подмену `cartId`, user id или route state.

## AC-009-10. Backend contract
- Backend предоставляет endpoints чтения, добавления, изменения количества, удаления, пересчета и validation основной корзины.
- Backend предоставляет endpoints чтения промо-предложений и применения offer к основной корзине.
- Backend предоставляет отдельные endpoints или явно типизированный contract для supplementary cart и supplementary shopping offers.
- Все endpoints отражены в OpenAPI owning module и используют DTO без hardcoded user-facing текста.
- Все предопределенные сообщения, которые могут попасть во frontend, оформлены mnemonic-кодами `STR_MNEMO_*`.

## AC-009-11. Frontend i18n
- Все новые user-facing строки корзины, промо, supplementary order, CTA, статусов, ошибок, aria-label и empty states вынесены в frontend i18n dictionaries.
- Новые ключи добавлены во все поддерживаемые frontend languages в рамках этой же задачи.
- React-компоненты не содержат новых hardcoded пользовательских строк.
- Backend mnemonic-коды feature #9 добавлены в поддерживаемые frontend dictionaries.

## AC-009-12. Адаптивность и доступность
- Desktop layout корзины показывает строки, блок итогов и промо-переходы без перекрытий и горизонтального скролла.
- Mobile layout позволяет менять количество, удалять строки, видеть итоги и перейти к промо/checkout одной колонкой.
- Quantity controls, remove actions, offer selection, tabs/segments основной корзины и supplementary order доступны с клавиатуры.
- Изменение количества, удаление строки и пересчет итогов объявляются доступным способом для assistive technologies.

## AC-009-13. Ошибки и устойчивость
- Ошибка загрузки корзины, ошибка пересчета, недоступность складских остатков, ошибка промо-правил и конфликт версии корзины имеют отдельные контролируемые состояния.
- Frontend не показывает пользователю stack trace, raw exception, raw JSON или технический текст.
- При временной интеграционной ошибке пользователь может повторить действие без потери текущей корзины.
- Optimistic UI не оставляет неконсистентное количество или итог, если backend отклонил изменение.

## AC-009-14. Безопасность и audit
- Все cart identifiers, offer identifiers и quantity values обрабатываются как untrusted input.
- Cart operations требуют авторизации согласно текущей security-модели проекта и проверяют принадлежность корзины пользователю или support permission.
- Audit trail фиксирует ключевые события: создание корзины, добавление, изменение количества, удаление, применение offer, validation failure и support action.
- Логи не содержат лишних персональных данных, закрытых token, полных session identifiers или пользовательских платежных данных.

## AC-009-15. Тестовое покрытие
- Managed API test начинается с логина подходящей роли и проверяет чтение пустой корзины, добавление товара, изменение количества, удаление, пересчет итогов и блокировку недоступного товара.
- Managed API test проверяет промо-предложения основной корзины, применение offer и отдельный supplementary cart для партнера.
- Managed UI test начинается с логина подходящей роли и проверяет `/cart`, `/cart/shopping-offers`, `/cart/supplementary`, `/cart/supplementary/shopping-offers`, quantity controls, удаление, промо и мобильную ширину.
- Managed end-to-end tests агрегируют реальный тест feature #9 вместе с предыдущими feature-тестами зеленого пути, а не содержат placeholder assertions.
- Runtime-копии managed tests синхронизированы из `agents/tests/` по `agents/tests/targets.yml` и содержат marker comment.

## Версионная база
Новые технологии не вводятся. Используются текущие версии существующего Spring Boot monolith, Java/Maven baseline, React/TypeScript web-shell и Ant Design-compatible UI-подход проекта. При реализации нельзя добавлять отдельный cart service, внешний promotion engine, новый frontend state manager или альтернативный checkout workflow без отдельного архитектурного решения и обновления workflow-артефактов.
