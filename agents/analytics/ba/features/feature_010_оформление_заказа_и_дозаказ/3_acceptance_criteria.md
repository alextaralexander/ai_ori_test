# Acceptance criteria. Feature 010. Оформление заказа и дозаказ

## AC-010-01. Маршруты оформления
- Маршрут `/order` доступен авторизованному покупателю или партнеру только при наличии валидной основной корзины.
- Маршрут `/order/supplementary` доступен партнеру только при наличии валидного supplementary cart и разрешенного права на дозаказ.
- Неавторизованный пользователь при открытии защищенных маршрутов получает локализованный сценарий входа с сохранением intended route.
- Основной checkout и supplementary checkout не смешивают cartId, строки, итоги, выгоды и бизнес-ограничения.

## AC-010-02. Старт checkout из корзины
- Checkout стартует только после server-side validation состава корзины, цен, доступности, campaign status, role permissions и benefit rules.
- Если корзина пуста или содержит blocking availability status, пользователь получает список причин и возврат к исправлению корзины.
- Checkout snapshot фиксирует состав, campaignId, тип заказа, subtotal, скидки, применимые выгоды и preliminary grand total.
- Повторный старт checkout для той же валидной корзины возвращает актуальный черновик оформления без создания дублей.

## AC-010-03. Получатель и контактные данные
- Пользователь может выбрать себя или другого разрешенного получателя согласно роли и правилам заказа.
- Имя, телефон, email и обязательные контактные поля валидируются на frontend и backend.
- Backend возвращает только structured validation errors и mnemonic-коды `STR_MNEMO_*` для предопределенных пользовательских сообщений.
- Ошибки контактов не сбрасывают уже выбранные адрес, доставку, оплату и выгоды, если эти данные остаются применимыми.

## AC-010-04. Адрес и пункт выдачи
- Пользователь может выбрать сохраненный адрес, создать новый адрес или выбрать пункт выдачи, если он доступен для способа доставки.
- Адрес валидируется по обязательным полям, региональным ограничениям, зонам доставки и возможности сборки заказа.
- Пункт выдачи содержит идентификатор, название, адрес, режим работы, доступность и ограничения по заказу.
- Подмена addressId или pickupPointId другого пользователя не позволяет оформить заказ.

## AC-010-05. Способы доставки
- Backend возвращает список доступных способов доставки для выбранного адреса, состава заказа, campaign context и типа заказа.
- Каждый способ доставки содержит код, стоимость, ожидаемый интервал, ограничения, признак доступности и structured reason при недоступности.
- Изменение адреса или состава заказа пересчитывает доступные доставки и итоговую стоимость.
- Недоступный способ доставки нельзя сохранить через подмену request body.

## AC-010-06. Оплата
- Пользователь может выбрать только платежные методы, доступные для роли, типа заказа, суммы к оплате и региона доставки.
- Платежная сессия создается с idempotency key и связана с checkout draft/order draft.
- Повторный запрос создания платежной сессии с тем же ключом не создает второй платеж и не меняет сумму сверх ожидаемого результата.
- Статусы `PENDING`, `PAID`, `FAILED`, `EXPIRED` и `CANCELLED` обрабатываются контролируемо и отражаются в состоянии заказа.

## AC-010-07. Электронный кошелек, кешбэк и выгоды
- Пользователь видит доступный баланс электронного кошелька, кешбэк и разрешенные каталожные выгоды без раскрытия закрытых финансовых данных.
- Примененная сумма кошелька не может превышать доступный баланс, сумму заказа и лимиты текущей кампании.
- Benefit-правила учитывают role segment, partner context, VIP/super order режим, retention-программы и тип заказа.
- Backend не применяет выгоды дважды при повторных запросах или повторном подтверждении.

## AC-010-08. Supplementary order
- Supplementary checkout использует отдельный checkout type `SUPPLEMENTARY` и не изменяет основной checkout пользователя.
- Для дозаказа проверяются partner status, право на supplementary order, campaignId, лимиты дозаказа, benefit-правила и складские остатки.
- Если дозаказ недоступен, пользователь видит локализованное контролируемое состояние и mnemonic-код причины.
- Применимые выгоды и доставка supplementary order рассчитываются отдельно от основной корзины.

## AC-010-09. VIP и super order режимы
- Checkout умеет принимать структурированные признаки VIP/super order режима, если они разрешены для пользователя и кампании.
- Запрещенный VIP/super order режим блокируется backend validation независимо от состояния frontend.
- Итоги, выгоды, доставка и доступные платежные методы пересчитываются с учетом режима.
- Причины недоступности режима передаются во frontend mnemonic-кодами `STR_MNEMO_*`.

## AC-010-10. Резервирование остатков
- Перед фиксацией заказа backend выполняет финальную проверку остатков и создает резерв через модульные контракты inventory/WMS-интеграции.
- Успешный резерв фиксирует зарезервированное количество для каждой строки заказа.
- Частичный резерв возвращает список строк, доступное количество и structured reason без создания неконсистентного заказа.
- При сбое резервирования пользователь может повторить действие или вернуться к корзине без потери checkout draft.

## AC-010-11. Подтверждение заказа
- Финальное подтверждение выполняет атомарную оркестрацию: validation, применение выгод, резерв, создание заказа, создание/проверку платежной сессии и публикацию событий.
- Подтверждение требует idempotency key; повтор с тем же ключом возвращает тот же результат, если первый запрос уже создал заказ.
- После успешного подтверждения пользователь видит номер заказа, статус, сумму, доставку и следующий шаг оплаты/ожидания.
- Нельзя подтвердить заказ со stale checkout version; пользователь получает контролируемый конфликт версии и актуальные данные.

## AC-010-12. Интеграции
- Интеграция WMS/1C используется для проверки остатков и резервов через устойчивый контракт без раскрытия технических ошибок пользователю.
- Интеграция платежной системы используется для создания платежной сессии и обработки callback/status polling.
- Интеграция доставки используется для получения способов доставки, пунктов выдачи и ограничений.
- Временная недоступность интеграции переводит checkout в контролируемое состояние retry/degraded без stack trace, raw exception или raw JSON.

## AC-010-13. Backend contract
- Backend предоставляет endpoints чтения checkout draft, обновления получателя, адреса, доставки, оплаты, применения выгод, validation, подтверждения заказа и чтения результата.
- Контракты явно различают основной заказ и supplementary order.
- DTO не содержат hardcoded user-facing текстов; предопределенные сообщения передаются mnemonic-кодами `STR_MNEMO_*`.
- Все endpoints отражены в OpenAPI owning module и соответствуют runtime Swagger.

## AC-010-14. Frontend i18n
- Все новые user-facing строки checkout, supplementary checkout, контактов, адресов, доставки, оплаты, выгод, ошибок, aria-label и empty/error states вынесены в frontend i18n dictionaries.
- Новые ключи добавлены во все поддерживаемые frontend languages в рамках этой же задачи.
- React-компоненты не содержат новых hardcoded пользовательских строк.
- Backend mnemonic-коды feature #10 добавлены в поддерживаемые frontend dictionaries.

## AC-010-15. Безопасность и audit
- Checkout identifiers, cart identifiers, address identifiers, pickup point identifiers, payment session identifiers и quantities обрабатываются как untrusted input.
- Все операции checkout требуют авторизации и проверяют принадлежность данных пользователю или support permission.
- Audit trail фиксирует старт checkout, изменение получателя, адреса, доставки, оплаты, применение выгод, validation failure, подтверждение, резерв, платежные события и support action.
- Логи не содержат полных платежных реквизитов, закрытых token, полных session identifiers или лишних персональных данных.

## AC-010-16. Адаптивность и доступность
- Desktop layout показывает шаги checkout, форму текущего шага, итог заказа и предупреждения без перекрытий и горизонтального скролла.
- Mobile layout показывает checkout одной колонкой с доступной навигацией по шагам, сохранением CTA и читабельными ошибками.
- Формы, radio/segmented controls, выбор адреса, доставки, оплаты, применения выгод и подтверждение доступны с клавиатуры.
- Ошибки валидации, пересчет итогов, изменение шага и результат подтверждения объявляются доступным способом для assistive technologies.

## AC-010-17. Тестовое покрытие
- Managed API test начинается с логина подходящей роли и проверяет старт checkout, выбор получателя, адреса, доставки, оплаты, применение кошелька/выгод, validation и подтверждение заказа.
- Managed API test проверяет supplementary checkout партнера, изоляцию от основного checkout и блокировку недоступного дозаказа.
- Managed API test проверяет idempotency подтверждения, частичный резерв, отказ платежа и запрет доступа к чужому checkout.
- Managed UI test начинается с логина подходящей роли и проверяет `/order`, `/order/supplementary`, пошаговый флоу, ошибки валидации, mobile layout и доступность ключевых controls.
- Managed end-to-end tests агрегируют реальный тест feature #10 вместе с предыдущими feature-тестами зеленого пути, а не содержат placeholder assertions.
- Runtime-копии managed tests синхронизированы из `agents/tests/` по `agents/tests/targets.yml` и содержат marker comment.

## Версионная база
Новые технологии не вводятся. Используются текущие версии существующего Spring Boot monolith, Java/Maven baseline, React/TypeScript web-shell и Ant Design-compatible UI-подход проекта. При реализации нельзя добавлять отдельный checkout service, внешний payment orchestrator, новый frontend state manager или альтернативный order workflow без отдельного архитектурного решения и обновления workflow-артефактов.
