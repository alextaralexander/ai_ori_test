# Acceptance criteria. Feature 011. История и детали заказов

## AC-011-01. Маршруты истории заказов
- Маршрут `/order/order-history` доступен только авторизованному покупателю или партнеру и показывает историю заказов текущего пользователя.
- Маршрут `/order/order-history/:orderId` доступен только владельцу заказа или сотруднику поддержки с явным правом support-доступа.
- Неавторизованный пользователь при открытии защищенных маршрутов получает локализованный сценарий входа с сохранением intended route.
- Подмена `orderId`, `customerId`, `partnerId` или параметров фильтра не раскрывает чужие заказы.

## AC-011-02. Список заказов
- Backend возвращает постраничный список заказов с номером, датой, campaignId, типом заказа, суммой, валютой, статусом заказа, оплатой, доставкой и кратким составом.
- Список сортируется по дате создания по убыванию по умолчанию.
- Пагинация содержит текущую страницу, размер страницы, общее количество элементов и признак наличия следующей страницы.
- Empty state отображается локализованно и не содержит технических сообщений.

## AC-011-03. Поиск и фильтры
- Пользователь может искать по номеру заказа, названию товара и артикулу.
- Пользователь может фильтровать историю по периоду, кампании, типу заказа `MAIN`/`SUPPLEMENTARY`, статусу заказа, статусу оплаты, статусу доставки и наличию претензий.
- Backend валидирует фильтры server-side и игнорирует или отклоняет недопустимые значения контролируемым образом.
- Состояние фильтров сохраняется при переходе из списка в детали и обратно.

## AC-011-04. Детальная карточка заказа
- Детали заказа содержат номер, дату, campaignId, тип заказа, канал создания, статус, итоговые суммы, валюту, примененные скидки, cashback, электронный кошелек и delivery/payment summary.
- Состав заказа содержит строки с productId, SKU, названием, количеством, ценой, скидками, подарочным признаком, доступностью повторного заказа и причиной ограничения.
- Детали показывают адрес доставки или пункт выдачи, получателя, masked contact fields и ожидаемые сроки.
- Пользователь не видит закрытые платежные реквизиты, служебные token или raw integration payload.

## AC-011-05. Статусы и события
- Детали заказа содержат timeline событий заказа: создание, validation, резервирование, оплата, сборка, передача в доставку, получение, отмена, претензия и сервисные изменения.
- Каждое событие содержит дату, тип, публичный статус, источник и mnemonic-код описания при необходимости.
- Статусы оплаты и доставки обновляются из согласованного backend state, а frontend не выводит hardcoded business text из API.
- Пользователь видит дату последнего обновления статуса.

## AC-011-06. Предупреждения и действия
- Заказ может возвращать structured warnings для ожидающей оплаты, просроченной оплаты, частичной сборки, задержки доставки, недоступного повторного заказа и ограничения претензии.
- Предопределенные предупреждения передаются mnemonic-кодами `STR_MNEMO_*` и локализуются на frontend.
- Действие повторного заказа доступно только для строк, которые разрешены текущей кампанией, ценами и остатками.
- Действие создания претензии доступно только для заказов и позиций, которые соответствуют правилам feature 012.

## AC-011-07. Повторный заказ
- При запуске repeat order backend проверяет текущую кампанию, актуальные цены, остатки, role permissions и ограничения supplementary order.
- Недоступные позиции не переносятся в корзину как валидные строки; пользователь видит структурированную причину.
- Повтор основного заказа не изменяет supplementary cart.
- Повтор supplementary order доступен только партнеру с разрешенным статусом и создает отдельный supplementary cart context.

## AC-011-08. Переход к оплате
- Для заказа с оплатой `PENDING` или `FAILED_RETRYABLE` пользователь видит действие продолжения оплаты, если платежная сессия еще валидна или может быть пересоздана идемпотентно.
- Для завершенной, отмененной или истекшей оплаты действие оплаты скрыто или заблокировано с локализованной причиной.
- Повторный переход к оплате не создает дубль заказа и не меняет сумму без повторной server-side validation.

## AC-011-09. Support-доступ
- Сотрудник поддержки может открыть детали заказа пользователя только через разрешенный backend endpoint или permission context.
- Support view показывает достаточные данные для помощи, но маскирует чувствительные контакты и платежные реквизиты.
- Каждый support-просмотр фиксируется в audit trail с userId сотрудника, orderId, временем и причиной доступа, если причина передана.
- Обычный пользователь не может использовать support-параметры для обхода ownership checks.

## AC-011-10. Backend contract
- Backend предоставляет endpoints чтения списка заказов, чтения деталей заказа, запуска repeat order и получения разрешенных действий.
- DTO используют структурированные enums для order/payment/delivery statuses и не содержат hardcoded user-facing text.
- Предопределенные пользовательские сообщения передаются mnemonic-кодами `STR_MNEMO_*`.
- OpenAPI owning module `order` описывает новые контракты и соответствует runtime Swagger для monolith module.

## AC-011-11. Frontend i18n и типизация
- Все новые user-facing строки истории заказов, фильтров, деталей, timeline, статусов, предупреждений, CTA, empty/error/loading states и aria-label вынесены в frontend i18n dictionaries.
- Новые i18n keys добавлены во все поддерживаемые frontend languages в рамках этой же задачи.
- React-компоненты не типизируют возвращаемое значение как `JSX.Element`; используется inference или `ReactElement`.
- Backend mnemonic-коды feature #11 добавлены в поддерживаемые frontend dictionaries.

## AC-011-12. Адаптивность и доступность
- Desktop layout показывает фильтры, список заказов и ключевые summary-поля без горизонтального скролла и перекрытий.
- Mobile layout показывает фильтры в компактном виде, карточки заказов одной колонкой и доступный переход в детали.
- Timeline, таблица/список товаров, CTA оплаты, repeat order и claim actions доступны с клавиатуры.
- Loading, error, empty и status update states объявляются доступным способом для assistive technologies.

## AC-011-13. Тестовое покрытие
- Managed API test начинается с логина подходящей роли и проверяет список истории, фильтры, поиск, детали заказа, timeline, предупреждения, repeat order и запрет чужого заказа.
- Managed API test проверяет partner supplementary order, pending payment action и support-доступ с audit expectation.
- Managed UI test начинается с логина подходящей роли и проверяет `/order/order-history`, `/order/order-history/:orderId`, фильтры, поиск, детали, mobile layout и доступность ключевых controls.
- Managed end-to-end tests агрегируют реальный тест feature #11 вместе с предыдущими feature-тестами зеленого пути, а не содержат placeholder assertions.
- Runtime-копии managed tests синхронизированы из `agents/tests/` по `agents/tests/targets.yml` и содержат marker comment.

## Версионная база
Новые технологии не вводятся. Используются текущие версии существующего Spring Boot monolith, Java/Maven baseline, React/TypeScript web-shell и Ant Design-compatible UI-подход проекта. Реализация должна оставаться в module `order`, использовать XML Liquibase changelog для feature #11 и не вводить отдельный order-history microservice, новый frontend state manager или альтернативный локализационный механизм без отдельного архитектурного решения.
