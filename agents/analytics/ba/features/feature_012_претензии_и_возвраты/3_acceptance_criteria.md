# Acceptance criteria. Feature 012. Претензии и возвраты

## AC-012-01. Доступ и маршруты
- `/order/claims/claim-create` доступен только авторизованному пользователю с ролью покупателя или партнера.
- `/order/claims/claims-history` показывает только претензии текущего пользователя, кроме разрешенного support-сценария.
- `/order/claims/claims-history/:claimId` возвращает 403 при попытке открыть чужую претензию без support-контекста.
- Если претензия не найдена, backend возвращает mnemonic `STR_MNEMO_ORDER_CLAIM_NOT_FOUND`.

## AC-012-02. Создание претензии
- Пользователь может создать претензию по существующему заказу с хотя бы одной выбранной позицией.
- Для каждой позиции фиксируются productCode, sku, quantity, причина проблемы и запрошенное решение.
- Поддерживаются решения `REFUND`, `REPLACEMENT`, `MISSING_ITEM`, `SERVICE_REVIEW` и `REJECTED` как machine-readable значения.
- Повторная отправка с тем же idempotency key не создает дублирующий кейс.

## AC-012-03. Валидация
- Нельзя создать претензию без номера заказа, выбранных позиций или причины.
- Нельзя запросить количество больше доступного количества в заказе.
- Нельзя создать претензию по позиции, у которой `claimAvailable=false`.
- Backend возвращает только mnemonic-коды `STR_MNEMO_*` и структурированные details без hardcoded user-facing текста.

## AC-012-04. Компенсация и решение
- Backend рассчитывает предварительную сумму компенсации по выбранным позициям и quantity.
- Детали претензии содержат requestedResolution, approvedResolution, refundAmount, currencyCode и nextAction.
- Для частичного решения сумма и статус рассчитываются на уровне позиции.
- Отказ всегда содержит public reason mnemonic, доступный frontend для локализации.

## AC-012-05. История претензий
- История возвращается постранично и содержит claimId, orderNumber, статус, тип решения, сумму, валюту и дату обновления.
- Поиск работает по claimId, orderNumber, productName и sku.
- Фильтры работают по статусу, типу решения и периоду.
- Empty state возвращает mnemonic `STR_MNEMO_ORDER_CLAIMS_EMPTY`.

## AC-012-06. Детали кейса
- Детали содержат выбранные позиции, вложения, публичные комментарии, системные события и доступные действия.
- История событий включает создание, назначение, складскую проверку, решение и завершение.
- Пользователь видит только публичные комментарии; внутренние комментарии оператора не попадают в клиентский контракт.
- Вложения возвращаются как метаданные, без передачи приватных storage paths.

## AC-012-07. Партнерский контекст
- Для партнера детали показывают влияние на business volume и бонусы как структурированные поля.
- Возврат по supplementary order не смешивается с основным заказом.
- В истории претензий партнер видит отдельный признак partnerImpact.

## AC-012-08. Support и логистика
- Support-роль может открыть кейс с audit context и признаком `auditRecorded=true`.
- Склад и логистика получают только поля, необходимые для проверки и возвратной маршрутизации.
- Все изменения статуса фиксируются в событиях кейса с sourceSystem и occurredAt.

## AC-012-09. Frontend i18n
- Все новые заголовки, кнопки, фильтры, статусы, placeholder, error и success-сообщения определены в `resources_ru.ts` и `resources_en.ts`.
- React-компоненты не содержат новых hardcoded user-facing строк.
- Backend mnemonic-коды, которые могут попасть во frontend, добавлены во все поддерживаемые словари.

## AC-012-10. UI и доступность
- Страница создания поддерживает клавиатурный выбор заказа, позиций, причины, решения и отправку.
- История и детали имеют стабильные `data-testid` для managed UI tests.
- Ошибки и успешные действия отображаются через Ant Design-compatible Alert и не перекрывают основной контент.
- На мобильной ширине фильтры истории остаются доступными без горизонтальной прокрутки.

## AC-012-11. Тесты и синхронизация
- Managed API test покрывает создание, историю, детали, валидацию, доступ и support-сценарий.
- Managed UI test покрывает создание претензии, поиск в истории, открытие деталей и обработку forbidden/error state.
- End-to-end managed tests агрегируют реальный FeatureApiTest и реальный feature UI flow, а не placeholder-assertions.
- Runtime-копии тестов создаются только из `agents/tests/` с marker comment.

## AC-012-12. Liquibase и пакетная структура
- Dedicated Liquibase XML для feature #12 создан в changelog order и не смешан с changelog других фич.
- Backend runtime code остается в `order/impl/controller` и `order/impl/service`, DTO в `order/api`, domain-структуры в `order/domain`.
- Новые тест-классы заканчиваются на `Test`.
