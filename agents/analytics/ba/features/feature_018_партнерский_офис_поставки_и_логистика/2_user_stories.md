# User stories feature 018. Партнерский офис, поставки и логистика

## Контекст
Фича восстанавливает партнерский офис Best Ori Gin как рабочий контур управления заказами, поставками, складскими перемещениями и логистикой для офлайн-продаж. Пользователь должен видеть не только отдельный заказ или поставку, но и связанный операционный процесс: какие заказы вошли в supply-поставку, где находится отгрузка, какие позиции приняты, какие отклонения обнаружены, какие действия доступны и какие downstream-процессы уже связаны с order, claim, pickup и delivery workflows.

## Роли
- Партнер-офис: отвечает за приемку supply-поставок, контроль заказов офлайн-точки, фиксацию недостач и работу со связанными доставками.
- Логистический оператор: сопровождает отгрузки, перемещения, статусы доставки, контроль комплектности и передачу данных в складской контур.
- Региональный менеджер: контролирует офисы региона, SLA поставок, проблемные отклонения, финансово-операционные показатели и эскалации.
- Система WMS/1C: передает сведения об остатках, движениях, отгрузках, приемке, недостачах и внешних идентификаторах складских документов.
- Система заказов, претензий, доставки и пунктов выдачи: предоставляет связанные order, claim, pickup и delivery данные, на которые партнерский офис должен переходить из логистического контура.

## Stories

### US-018-01. Список всех заказов партнерского офиса
Как партнер-офис, я хочу открыть `/partner-office/all-orders` и увидеть все заказы моего офиса с номером, клиентом или партнером, каталогом, статусом оплаты, статусом сборки, supply-связкой и delivery-состоянием, чтобы контролировать операционную нагрузку.

### US-018-02. Фильтрация заказов офиса
Как партнер-офис, я хочу фильтровать заказы по периоду, каталогу, статусу, supplyId, delivery method, pickup point, клиенту, партнеру и признаку отклонения, чтобы быстро находить заказы, требующие обработки.

### US-018-03. Сводный отчет партнерского офиса
Как региональный менеджер, я хочу открыть `/partner-office/report` и увидеть показатели по заказам, supply-поставкам, приемке, недостачам, SLA и проблемным маршрутам, чтобы управлять качеством логистики по офисам региона.

### US-018-04. Список supply-поставок
Как партнер-офис, я хочу открыть `/partner-office/supply` и увидеть supply-поставки с датой плановой отгрузки, датой прибытия, статусом, количеством заказов, коробов, SKU, недостач и отклонений, чтобы планировать приемку.

### US-018-05. Детали supply-поставки
Как партнер-офис, я хочу открыть `/partner-office/supply/:supplyId`, чтобы увидеть состав поставки, связанные заказы, складские документы, движения, историю статусов и доступные действия по приемке.

### US-018-06. Заказ внутри поставки
Как партнер-офис, я хочу открыть `/partner-office/supply/orders/:orderId` из карточки поставки, чтобы проверить состав конкретного заказа, упаковочные места, статусы комплектности, delivery-связку и доступные действия.

### US-018-07. Контроль комплектности и недостач
Как партнер-офис, я хочу фиксировать обнаруженные недостачи, излишки, повреждения и расхождения по позициям, чтобы дальнейшая претензия или корректировка создавалась на основании структурированных данных приемки.

### US-018-08. История складских движений
Как логистический оператор, я хочу видеть историю складских движений, внешние WMS/1C номера, статусы отгрузки, приемки и перемещения, чтобы расследовать, где возникло отклонение.

### US-018-09. Управление статусом поставки
Как логистический оператор, я хочу переводить поставку через разрешенные состояния `PLANNED`, `IN_TRANSIT`, `ARRIVED`, `ACCEPTANCE_IN_PROGRESS`, `ACCEPTED`, `PARTIALLY_ACCEPTED`, `BLOCKED`, чтобы логистический процесс оставался воспроизводимым.

### US-018-10. Переходы в связанные workflows
Как партнер-офис, я хочу переходить из заказа или поставки к связанным order, claim, pickup и delivery workflows, чтобы не искать вручную претензию, пункт выдачи или доставку по номеру заказа.

### US-018-11. Эскалация проблемной поставки
Как региональный менеджер, я хочу видеть проблемные supply-поставки и создавать эскалацию с reason code, комментарием и ответственным, чтобы недостачи и нарушения SLA не оставались без владельца.

### US-018-12. Отчетность по SLA и отклонениям
Как региональный менеджер, я хочу видеть SLA отгрузки, транзита, приемки и обработки отклонений, чтобы оценивать работу склада, доставки и партнерских офисов.

### US-018-13. Защита доступа по офису и региону
Как пользователь платформы, я хочу видеть только те заказы, поставки и отчеты, которые разрешены моей роли, офису и региону, чтобы данные другого офиса не раскрывались через прямой URL.

### US-018-14. Аудит логистических действий
Как владелец платформы, я хочу, чтобы изменения статусов, фиксация отклонений, комментарии, эскалации и переходы между workflows сохраняли audit metadata, correlation id и внешний source reference, чтобы расследование можно было повторить.

### US-018-15. Локализация и backend-frontend message contract
Как frontend-пользователь, я хочу видеть локализованные статусы, ошибки, подсказки и результаты действий, а backend должен передавать предопределенные пользовательские сообщения только mnemonic-кодами `STR_MNEMO_*`, чтобы интерфейс оставался единообразным на всех поддерживаемых языках.

## Версионный baseline на 27.04.2026
- Java 26.0.1 является актуальным Oracle JDK на дату старта, но текущий backend monolith зафиксирован на Java 25 и `maven.compiler.release=25`; реализация feature #18 не повышает runtime и должна быть совместима с этим утвержденным baseline до отдельной задачи миграции Java 26.
- Spring Boot 4.0.6 используется как текущая latest stable линия для backend monolith.
- Frontend web-shell использует зависимости `latest` для React 19.2, TypeScript 6.0, Ant Design 5.x и Vite stable; feature #18 не вводит новых frontend runtime-технологий.
- Liquibase changesets для backend-части должны быть XML в owning module `partner-office`.
