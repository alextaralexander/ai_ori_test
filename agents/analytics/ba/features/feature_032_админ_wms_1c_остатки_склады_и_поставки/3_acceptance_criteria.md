# Acceptance criteria. Feature 032. Админ: WMS/1C, остатки, склады и поставки

## Общие критерии
1. Административный пользователь с разрешением на WMS-контур может открыть раздел остатков, складов, поставок, перемещений, резервов и обменов WMS/1C; пользователь без разрешения получает отказ доступа без раскрытия складских данных.
2. Все операции фильтрации, просмотра, создания, изменения статуса, ручной корректировки и повторной синхронизации сохраняют actor, action code, source, correlationId, timestamp и business payload summary в audit trail.
3. Backend не возвращает во frontend предопределенные пользовательские сообщения как текст; для ошибок, предупреждений и результатов действий используются только mnemonic-коды `STR_MNEMO_ADMIN_WMS_*`.
4. Все новые frontend-строки, статусы, фильтры, кнопки, подсказки и пустые состояния вынесены в `resources_ru.ts` и `resources_en.ts`; hardcoded user-facing text в React-компонентах отсутствует.
5. Все новые backend runtime classes размещены в owning module по пакетной политике `api/domain/db/impl`, а конкретные классы внутри `impl` распределены по role subpackages `controller`, `service`, `mapper`, `validator`, `config`, `event`, `scheduler` или аналогичным.
6. Все Liquibase changesets созданы в отдельном XML changelog feature #32 внутри owning module `db` package и не дописаны в общий накопительный changelog.

## Склады
1. Список складов отображает warehouse id, название, тип, регион, статус, source system, external warehouse id, каналы продаж, SLA приемки, SLA отгрузки, время последней синхронизации и признак проблемного обмена.
2. Список складов фильтруется по статусу, региону, типу, source system, external id, каналу продаж и наличию ошибок синхронизации; фильтры не сбрасываются при возврате из карточки.
3. Карточка склада показывает зоны хранения, каналы, связанные точки выдачи, ограничения продаж, расписание синхронизации, последние документы WMS/1C и историю изменения статуса.
4. Смена статуса склада разрешена только по допустимым transition rules и требует reason code для блокировки, maintenance и archive.
5. Если склад переведен в `TEMPORARILY_BLOCKED` или `MAINTENANCE`, новые sellable reservations с этого склада не создаются, но существующие документы и история остаются доступными.

## Остатки и доступность SKU
1. Сводка остатков показывает SKU, товар, catalog period, warehouse, sellable quantity, reserved quantity, blocked quantity, inbound quantity, backorder quantity, available-to-promise и время последнего обновления.
2. Остатки фильтруются по SKU, товару, складу, каталогу, каналу продаж, наличию резерва, availability status и источнику данных.
3. Availability status рассчитывается из физического остатка, резервов, блокировок, inbound-поставок и policy `SELLABLE`, `RESERVED_ONLY`, `BACKORDER_ALLOWED`, `NOT_SELLABLE`, `QUALITY_HOLD`.
4. Система не допускает отрицательный `sellable quantity`, если для SKU и канала не включена явная backorder policy.
5. При изменении availability policy каталог и оформление заказа получают обновленные структурированные данные без ручной публикации frontend-кода.
6. Для SKU отображается история движений с типом операции, количеством, документом, источником, external reference и correlationId.

## Поставки и перемещения
1. Поставка создается с номером, складом-получателем, source system, supplier, ожидаемыми датами, списком SKU, плановым количеством и external document reference.
2. Поставка проходит только через разрешенные статусы `PLANNED`, `CONFIRMED`, `IN_TRANSIT`, `ARRIVED`, `ACCEPTANCE_IN_PROGRESS`, `ACCEPTED`, `PARTIALLY_ACCEPTED`, `CANCELLED`.
3. Приемка поставки фиксирует плановое количество, фактическое количество, повреждения, недостачи, излишки, reason code и документ-основание по каждой позиции.
4. После подтверждения приемки остатки увеличиваются один раз; повторное подтверждение того же external document id не удваивает stock movement.
5. Частично принятая поставка создает discrepancy records и помечает позиции, требующие претензии, корректировки или повторной проверки.
6. Межскладское перемещение хранит склад-источник, склад-получатель, состав SKU, плановое количество, фактическое количество, внешние документы отправки и приемки.
7. Отмена поставки или перемещения возможна только до момента финальной приемки и не удаляет историю уже созданных движений.

## Резервы, заказы и частичная отгрузка
1. Резерв создается с orderId, SKU, warehouseId, quantity, hold expiration, status и source channel; повторный запрос с тем же idempotency key не создает дубль.
2. Резерв может перейти в `HELD`, `CONFIRMED`, `RELEASED`, `EXPIRED`; запрещены переходы, которые освобождают больше товара, чем было зарезервировано.
3. Если товара недостаточно, backend возвращает структурированный результат с доступным количеством, blocked reason, backorder possibility и mnemonic-кодом для frontend.
4. Частичная отгрузка фиксирует недоступные позиции, доступные позиции, reason code и связь с order workflow, чтобы администратор заказов мог предложить дозаказ, backorder или отмену части заказа.
5. Release reservation при отмене заказа или истечении hold возвращает количество в sellable stock только один раз.

## Интеграция WMS/1C
1. Источник синхронизации хранит source type `WMS`, `ONE_C`, `MANUAL_IMPORT`, статус, расписание, endpoint metadata без секретов, поддерживаемые типы сообщений и последние результаты обмена.
2. Ручной запуск синхронизации доступен по складу, SKU, периоду и типу документа; запуск получает correlationId и отображается в журнале обмена.
3. Входящие сообщения обрабатываются идемпотентно по `sourceSystem`, `externalDocumentId`, `externalLineId` и `idempotencyKey`.
4. Дубли входящих сообщений не создают повторные движения остатков и фиксируются как безопасно проигнорированные или обновленные в журнале обмена.
5. Сообщения с конфликтом подтвержденных движений попадают в quarantine status и не меняют sellable stock до ручного решения оператора.
6. Ошибки обмена показывают статус, попытку, retry policy, payload checksum, reason code, affected warehouse/SKU/document и безопасное действие повтора.
7. Payload с персональными данными или секретами не сохраняется в audit trail; сохраняются только технические references, checksum и business summary.

## Расхождения и аудит
1. Расхождение создается по приемке, отгрузке, ручной корректировке или синхронизации и содержит type, severity, SKU, warehouse, quantity delta, source document, reason code, status и owner.
2. Расхождение можно перевести в `OPEN`, `IN_REVIEW`, `RESOLVED`, `REJECTED`, `ESCALATED`; закрытие требует resolution comment и audit trail.
3. Ручная корректировка остатка невозможна без reason code, документа-основания и разрешения пользователя на корректировки выбранного склада.
4. Аудит позволяет искать события по warehouseId, skuId, supplyId, externalDocumentId, actorUserId, actionCode, date range и correlationId.
5. В audit trail видны старые и новые business-значения, но не отображаются секреты интеграции и лишние персональные данные.

## UI и UX админки
1. Раздел админки содержит рабочие представления: склады, остатки, поставки, перемещения, резервы, расхождения, обмен WMS/1C и аудит.
2. Таблицы имеют фильтры, сортировку, пагинацию, пустые состояния, loading/error states и действия, доступные только по permissions.
3. Карточки склада, SKU, поставки и документа обмена открываются из списков без потери фильтров.
4. Опасные действия, влияющие на остатки или синхронизацию, требуют подтверждения, reason code и показывают результат через локализованный mnemonic.
5. UI не показывает пользователю raw payload WMS/1C по умолчанию; технический payload доступен только роли интеграционного оператора и маскирует секреты.

## Наблюдаемость и качество
1. Для синхронизации публикуются метрики количества обработанных сообщений, дублей, ошибок, retry, quarantine, задержки обработки и затронутых SKU.
2. API и UI тесты покрывают green path складов, остатков, поставок, резервов, интеграционного повтора и quarantine conflict.
3. Managed tests создаются сначала в `agents/tests/`, затем синхронизируются в runtime locations по `agents/tests/targets.yml` с marker comment.
4. End-to-end managed tests агрегируют реальные feature tests, а не содержат фиктивные проверки по имени feature или строковым литералам.
