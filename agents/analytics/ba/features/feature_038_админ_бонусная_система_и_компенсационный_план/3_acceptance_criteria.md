# Acceptance criteria feature #38. Админ: бонусная система и компенсационный план

## Управление правилами начисления
- Администратор бонусной программы может создать правило начисления с типом `ORDER_BONUS`, `STRUCTURE_LEVEL_BONUS`, `PROMO_BONUS` или `MANUAL_ADJUSTMENT`, указав период действия, приоритет, валюту, ставку, базу расчета, ограничения по каталогу, региону и партнерскому сегменту.
- Система не допускает публикацию правила, если период действия пустой, ставка отрицательная, валюта не задана, приоритет конфликтует с активным правилом того же типа или отсутствует хотя бы одно обязательное условие расчета.
- Правило проходит жизненный цикл `DRAFT -> ACTIVE -> SUSPENDED -> ARCHIVED`; удаление опубликованных правил запрещено, вместо него используется архивирование.
- Перед активацией система показывает preview расчета на тестовом заказе или выбранной партнерской ветке без записи финансовых транзакций.

## Компенсационный план и квалификации
- MLM-менеджер может создать квалификацию с кодом, названием, уровнем, периодом оценки, требованиями к личному объему, групповому объему, активности и глубине структуры.
- Система валидирует уникальность кода квалификации, непротиворечивость порогов и наличие связанной бонусной ставки для каждой активной ступени.
- Изменение активной квалификации создает новую версию условий с датой вступления в силу, а прошлые расчеты остаются связанными с версией, действовавшей в расчетном периоде.
- В карточке компенсационного плана видны активные правила, квалификации, версии, даты действия и зависимости от фич 014, 015, 016 и 017.

## Расчет начислений
- MLM-менеджер может запустить расчет за закрытый или текущий расчетный период; повторный запуск для того же периода требует явного режима recalculation и сохраняет ссылку на предыдущий расчет.
- Расчет формирует записи начислений со статусами `HOLD`, `ACCRUAL`, `PAYOUT_READY`, `REVERSED` или `ADJUSTED`, содержит partnerId, periodId, sourceOrderId/sourceTransactionId, ruleCode, amount, currency и correlationId.
- Возвраты, претензии, отмены заказов и ручные корректировки учитываются как reversal или adjustment, не меняя исходную финансовую запись задним числом.
- Backend не передает во frontend предопределенные пользовательские сообщения текстом; для ошибок и статусов, которые должны отображаться пользователю, используется формат `STR_MNEMO_*`.

## Payout batches и финансовая сверка
- Финансовый менеджер может сформировать payout batch по периоду, валюте, региону и партнерскому сегменту только из начислений в статусе `PAYOUT_READY`.
- Payout batch проходит статусы `DRAFT`, `APPROVED`, `SENT`, `PAID`, `REJECTED`, `CANCELLED`; переходы статусов доступны только ролям с соответствующими правами.
- Отправка payout batch во внешнюю bonus/payment-систему идемпотентна: повторная отправка с тем же externalId не создает дублей выплат.
- Финансовый менеджер может выгрузить CSV/XLSX/PDF-сверку начислений, удержаний, корректировок и выплат; персональные данные маскируются в административных выгрузках по правилам безопасности платформы.

## Интеграции и эксплуатация
- Интеграционный администратор может видеть статус последнего обмена с внешней bonus-системой: направление, endpoint alias, retryCount, checksum, lastErrorCode, lastErrorMessageMnemonic и correlationId.
- При ошибке интеграции система сохраняет техническую причину без секретов, показывает frontend только mnemonic-код и позволяет повторить безопасные сообщения.
- При недоступности внешней bonus-системы администратор может переключить расчет в режим внутреннего движка, если это разрешено эксплуатационной настройкой.

## Аудит и безопасность
- Каждое создание, изменение, активация, приостановка, архивирование правила, запуск расчета, корректировка и изменение payout batch попадает в аудит с actorId, role, actionCode, entityType, entityId, before/after summary, createdAt и correlationId.
- Пользователь без административной роли бонусной программы, финансовой роли или MLM-роли не может открыть административные экраны и endpoint-ы фичи 038.
- Все новые frontend строки вынесены в i18n dictionaries для всех поддерживаемых языков; React-компоненты не содержат hardcoded user-facing text.
- Все новые backend runtime-классы размещены в `impl/<role>` подпакетах, JPA entities и repositories находятся только в `domain`, Liquibase changelog создан XML-файлом в `db` пакете owning module.

## Готовность к приемке
- Созданы BA, SA, архитектурные, тестовые и end-to-end артефакты workflow без пустых placeholder-файлов.
- Managed API и UI тесты начинаются с логина пользователя с соответствующей ролью и покрывают зеленые сценарии настройки правила, расчета начислений и формирования payout batch.
- Runtime-копии managed tests синхронизированы из `agents/tests/` по `agents/tests/targets.yml` и содержат marker comment о generated artifact.
