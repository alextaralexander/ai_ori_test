# Architecture description

Архитектура Best Ori Gin остается domain-oriented monolith с frontend web-shell и backend monolith-app. Feature #19 добавляет модуль `employee`, который обслуживает employee workspace, операторское создание заказа и поддержку проблемного заказа. Feature #20 расширяет тот же модуль операторской историей заказов. Feature #21 добавляет employee-претензии. Feature #22 добавляет employee-карточку партнера и отчет истории заказов партнера. Feature #23 добавляет employee profile settings и контролируемый elevated mode для повышенных прав сотрудника. Feature #24 добавляет модуль `auth`, который владеет session context, role routing, invitation code, active partner и controlled impersonation. Feature #25 добавляет модуль `platform-experience`, который владеет runtime config, consent preferences, notification preferences, analytics diagnostics и i18n diagnostics. Feature #26 добавляет модуль `admin-rbac`, который владеет internal accounts, roles, permission sets, responsibility scopes, security policies, service accounts и immutable audit trail. Feature #27 добавляет модуль `admin-cms`, который владеет административным lifecycle контента: материалы, версии, structured blocks, SEO, документы, review, publication schedule, public projection и audit trail. Feature #28 добавляет модуль `admin-referral`, который владеет административными landing variants, registration funnels, referral codes, attribution policy, conversion analytics и audit trail.

## Backend modules and links
`auth` является владельцем runtime session state, role policies, invitation code state, active partner state, partner search audit и impersonation session. `admin-rbac` является владельцем административной модели внутренних доступов: учетные записи сотрудников, роли, permission sets, responsibility scopes, password/MFA/session policies, service accounts и audit events. `admin-cms` является владельцем административного CMS-контура: материалы, версии, blocks, SEO metadata, document metadata, review decisions, publication windows, relations и audit events. `admin-referral` является владельцем административного referral-контура: landing variants, landing versions, structured landing blocks, registration funnel versions, referral codes, attribution policy, attribution events, conversion events и audit events. `platform-experience` является владельцем сквозного experience-state: consent preferences, notification preferences, runtime config snapshots, analytics adapter diagnostics и i18n missing key diagnostics. `employee` не нарушает владение существующих доменов: `order` остается владельцем заказов, `cart` - корзины, `profile` - персональных и партнерских данных, `authorization` - identity/roles/MFA/locks, `partner-office` - поставок, payment/WMS/delivery/bonus системы остаются интеграционными источниками. Employee хранит support-контекст, ссылки, order-history read-model, employee claims read/write model, partner-card audit, partner-report snapshot, employee profile settings, elevated requests/sessions и audit trail действий сотрудников.

- `auth` вызывает `authorization` для проверки роли, route policy, MFA/elevated policy и блокировок.
- `auth` вызывает `employee` для проверки active elevatedSessionId при controlled impersonation.
- `auth` вызывает `admin-rbac` для получения session permission matrix и завершения затронутых сессий после block/deactivation.
- `auth` пишет sessions, roles, route policies, invitation code state, active partner, impersonation sessions и audit events в Postgres через Liquibase XML managed schema.
- `admin-rbac` вызывает `authorization` для role, permission set, scope и MFA/elevated policy checks.
- `admin-rbac` вызывает `auth` для чтения current session context и termination affected sessions при block/deactivation/emergency deactivation.
- `admin-rbac` пишет internal accounts, roles, permission sets, role bindings, account assignments, responsibility scopes, security policies, service accounts, service secret rotations и audit events в Postgres через Liquibase XML managed schema.
- `admin-rbac` ограничивает service account scopes для WMS/1C, payment, delivery и bonus integrations; протокол внешнего вызова остается у соответствующей интеграционной системы, RBAC выдает только технический scope и credential validation.
- `admin-cms` вызывает `authorization` и `admin-rbac` для проверки role, permission set, responsibility scope и elevated policy на каждом административном CMS endpoint-е.
- `admin-cms` пишет CMS materials, versions, content blocks, SEO metadata, relations, review decisions, publication windows и audit events в Postgres через Liquibase XML managed schema.
- `admin-cms` проверяет fileReferenceId документов и изображений через S3/MinIO abstraction и не возвращает private storage paths во frontend.
- `admin-cms` передает в `public-content` только published render model для главной, новостей, FAQ, документов, offer pages и home blocks; draft/review/archived versions не доступны публичному frontend.
- `admin-referral` вызывает `authorization` и `admin-rbac` для проверки role, permission set, responsibility scope и elevated policy на каждом административном referral endpoint-е.
- `admin-referral` пишет landing variants, landing versions, landing blocks, registration funnels, funnel versions, referral codes, attribution policies, attribution events, conversion events и audit events в Postgres через Liquibase XML managed schema.
- `admin-referral` передает в `public-content` только active benefit landing render model и campaign context; draft, paused, archived и future scheduled variants не доступны публичному frontend.
- `admin-referral` интегрируется с `partner-onboarding` по HTTP/JSON для validation referral code, сохранения campaign/referral context и sponsor attribution при регистрации feature #008.
- `admin-referral` передает `platform-experience` и frontend analytics layer только consent-aware conversion events и machine-readable metrics без секретов, токенов или лишних персональных данных.
- `platform-experience` пишет consent preferences, notification preferences, runtime config snapshots, analytics diagnostics и i18n missing key diagnostics в Postgres через Liquibase XML managed schema.
- Frontend `AnalyticsProvider` отправляет consent-aware pageview/conversion events во внешние Yandex Metrika, Mindbox и Hybrid Pixel adapters; запрет consent блокирует отправку в соответствующий внешний канал.
- `employee` вызывает `authorization` для проверки роли, scope, MFA, блокировок и supervisor-доступа.
- `employee` читает partner profile/KPI из `profile` и `partner-reporting`, бонусный баланс и volume из `bonus-wallet`/bonus engine, order links и агрегированные статусы из `order`, payment event ids/statuses из payment, claim ids из claims, WMS batch/events из WMS/1C и delivery tracking из delivery.
- `employee` проверяет `fileReferenceId` employee-документов через S3/MinIO abstraction и не хранит приватные storage paths в API.
- `employee` передает `elevatedSessionId` и `correlationId` в связанные order, claim, partner, delivery и support flows как audit context для расширенных операций.
- `employee` пишет support/order-history/employee-claims/partner-card/profile/elevated read models и audit tables в Postgres через Liquibase XML managed schema.

## Backend package ownership
`com.bestorigin.monolith.auth.api` содержит DTO и REST-контракты auth module. `auth/domain` содержит JPA entities и repository interfaces. `auth/db` содержит Liquibase marker package и XML changesets, включая отдельный changeset feature #24. `auth/impl/controller`, `auth/impl/service`, `auth/impl/config`, `auth/impl/security`, `auth/impl/validator`, `auth/impl/event`, `auth/impl/exception` содержат runtime-логику. Новые runtime-классы не размещаются в root `impl`.

`com.bestorigin.monolith.employee.api` содержит DTO и REST-контракты. `domain` содержит JPA entities и repository interfaces. `db` содержит Liquibase marker package и XML changesets, включая отдельный changeset feature #23. `impl/controller`, `impl/service`, `impl/security`, `impl/validator`, `impl/mapper`, `impl/event`, `impl/exception` содержат runtime-логику. Новые runtime-классы не размещаются в root `impl`.

`com.bestorigin.monolith.platformexperience.api` содержит DTO и REST-контракты platform-experience module. `platformexperience/domain` содержит JPA entities и repository interfaces для consent, notification, analytics diagnostics, i18n diagnostics и runtime config snapshots. `platformexperience/db` содержит Liquibase marker package и отдельный XML changeset feature #25. `platformexperience/impl/controller`, `impl/service`, `impl/config`, `impl/validator`, `impl/mapper`, `impl/exception` содержат runtime-логику. Новые runtime-классы не размещаются в root `impl`.

`com.bestorigin.monolith.adminrbac.api` содержит DTO и REST-контракты admin-rbac module. `adminrbac/domain` содержит JPA entities и repository interfaces для internal accounts, roles, permission sets, assignments, responsibility scopes, security policies, service accounts, secret rotations и audit events. `adminrbac/db` содержит Liquibase marker package и отдельный XML changeset feature #26. `adminrbac/impl/controller`, `impl/service`, `impl/security`, `impl/validator`, `impl/mapper`, `impl/audit`, `impl/config`, `impl/exception` содержат runtime-логику. Новые runtime-классы не размещаются в root `impl`.

`com.bestorigin.monolith.admincms.api` содержит DTO и REST-контракты admin-cms module. `admincms/domain` содержит JPA entities и repository interfaces для CMS materials, versions, blocks, SEO, relations, review decisions и audit events. `admincms/db` содержит Liquibase marker package и отдельный XML changeset feature #27. `admincms/impl/controller`, `impl/service`, `impl/validator`, `impl/mapper`, `impl/audit`, `impl/config`, `impl/exception` содержат runtime-логику. Новые runtime-классы не размещаются в root `impl`.

`com.bestorigin.monolith.adminreferral.api` содержит DTO, REST-контракты и enum-контракты admin-referral module. `adminreferral/domain` содержит JPA entities и repository interfaces для landing variants, landing versions, landing blocks, registration funnels, funnel versions, referral codes, attribution policies/events, conversion events и audit events. `adminreferral/db` содержит Liquibase marker package и отдельный XML changeset feature #28. `adminreferral/impl/controller`, `impl/service`, `impl/validator`, `impl/mapper`, `impl/audit`, `impl/config`, `impl/exception` содержат runtime-логику. Новые runtime-классы не размещаются в root `impl`.
## Swagger and message contract
Swagger groups формируются через module metadata: `/v3/api-docs/auth` и `/swagger-ui/auth` для auth, `/v3/api-docs/employee` и `/swagger-ui/employee` для employee, `/v3/api-docs/platform-experience` и `/swagger-ui/platform-experience` для platform-experience, `/v3/api-docs/admin-rbac` и `/swagger-ui/admin-rbac` для admin-rbac, `/v3/api-docs/admin-cms` и `/swagger-ui/admin-cms` для admin-cms, `/v3/api-docs/admin-referral` и `/swagger-ui/admin-referral` для admin-referral. Backend возвращает только mnemonic-коды `STR_MNEMO_AUTH_*`, `STR_MNEMO_EMPLOYEE_*`, `STR_MNEMO_PLATFORM_*`, `STR_MNEMO_ANALYTICS_*`, `STR_MNEMO_ADMIN_RBAC_*`, `STR_MNEMO_ADMIN_CMS_*` и `STR_MNEMO_ADMIN_REFERRAL_*` для предопределенных сообщений. Frontend локализует их через поддерживаемые `resources_*.ts` dictionaries.

## Frontend
Frontend использует platform providers `I18nProvider`, `NotificationProvider`, `OfflineStatusProvider`, `AnalyticsProvider`, `ConsentPanel` и `AnalyticsDiagnostics UI`, а также auth components/hooks: `AuthProvider`, `PrivateRoute`, `ProfileRoute`, `EmployeeRoute`, `useInvitationCode`, `useActivePartner`, `usePartnerSearch`, `useSuperUserMode`, `useImpersonate`, `ActivePartnerSwitcher` и `ImpersonationPanel`. Employee views остаются: `EmployeeWorkspaceView`, `EmployeeOrderHistoryView`, employee claims UI, `EmployeePartnerCardView`, `EmployeePartnerReportView`, `EmployeeProfileSettingsView`, `EmployeeProfileSettingsGeneralView`, `EmployeeProfileSettingsContactsView`, `EmployeeProfileSettingsAddressesView`, `EmployeeProfileSettingsDocumentsView`, `EmployeeProfileSettingsSecurityView` и `EmployeeSuperUserView`. Feature #26 добавляет `AdminRbacView`, `AdminRbacAccountTable`, `AdminRbacAccountForm`, `AdminRbacRoleAssignment`, `AdminRbacPermissionPreview`, `AdminRbacSecurityPolicy`, `AdminRbacServiceAccountsView`, `AdminRbacAuditView` и `AdminRbacEmergencyDeactivate`. Feature #27 добавляет `AdminCmsView`, `AdminCmsMaterialTable`, `AdminCmsEditor`, `AdminCmsReviewQueue`, `AdminCmsPreview`, `AdminCmsVersionList`, `AdminCmsPublicationSchedule`, `AdminCmsSeoPanel` и `AdminCmsAuditTable`. Feature #28 добавляет `AdminReferralView`, `AdminReferralLandingTable`, `AdminReferralLandingEditor`, `AdminReferralFunnelEditor`, `AdminReferralCodeTable`, `AdminReferralAttributionPolicy`, `AdminReferralConversionReport` и `AdminReferralAuditTable`. Маршруты auth feature включают `/test-login`, protected route restore, invitation routes, active partner switch и impersonation panel. Feature #25 добавляет platform shell behavior для notification root, offline popup, reconnect notification, language switcher, consent panel и analytics diagnostics. Все user-facing строки находятся в i18n dictionaries; компоненты не типизируют return как `JSX.Element`.

## Infrastructure
Feature #28 не требует новых runtime infrastructure components сверх существующих Postgres/Liquibase XML, monolith deployment и frontend web-shell. Используются существующие observability/logging каналы и consent-aware analytics layer. Если deployment manifests содержат Swagger route exposure для module groups, admin-referral endpoints попадают в `/v3/api-docs/admin-referral` и `/swagger-ui/admin-referral` без отдельного ingress.

## Version baseline
Версионный baseline на 27.04.2026: Java 25, Spring Boot 4.0.6, Maven monolith, TypeScript/React/Ant Design web-shell с зависимостями frontend-проекта, Liquibase XML, PostgreSQL target schema, S3/MinIO document abstraction, Docker/Kubernetes stack без изменений в этом инкременте. Feature #28 не вводит новый технологический стек и не требует понижения версий.
