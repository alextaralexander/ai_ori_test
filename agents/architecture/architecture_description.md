# Architecture description

Архитектура Best Ori Gin остается domain-oriented monolith с frontend web-shell и backend monolith-app. Feature #19 добавляет модуль `employee`, который обслуживает employee workspace, операторское создание заказа и поддержку проблемного заказа. Feature #20 расширяет тот же модуль операторской историей заказов. Feature #21 добавляет employee-претензии. Feature #22 добавляет employee-карточку партнера и отчет истории заказов партнера. Feature #23 добавляет employee profile settings и контролируемый elevated mode для повышенных прав сотрудника. Feature #24 добавляет модуль `auth`, который владеет session context, role routing, invitation code, active partner и controlled impersonation. Feature #25 добавляет модуль `platform-experience`, который владеет runtime config, consent preferences, notification preferences, analytics diagnostics и i18n diagnostics. Feature #26 добавляет модуль `admin-rbac`, который владеет internal accounts, roles, permission sets, responsibility scopes, security policies, service accounts и immutable audit trail. Feature #27 добавляет модуль `admin-cms`, который владеет административным lifecycle контента: материалы, версии, structured blocks, SEO, документы, review, publication schedule, public projection и audit trail. Feature #28 добавляет модуль `admin-referral`, который владеет административными landing variants, registration funnels, referral codes, attribution policy, conversion analytics и audit trail. Feature #29 добавляет модуль `admin-pim`, который владеет административными категориями, товарами, брендами, атрибутами, тегами, медиа, рекомендациями, import/export jobs и audit trail. Feature #30 добавляет модуль `admin-catalog`, который владеет 21-дневными catalog campaigns, PDF-выпусками, страницами, hotspots, freeze window, rollover jobs, archive projection и audit trail. Feature #31 добавляет модуль `admin-pricing`, который владеет price lists, базовыми ценами, promo prices, segment rules, promotions, shopping offers, bundles, gift rules, threshold benefits, import/export jobs, публикацией и audit trail. Feature #32 добавляет модуль `admin-wms`, который владеет складами, остатками, availability rules, поставками, резервами, sync runs/messages, quarantine, расхождениями и audit trail. Feature #33 добавляет модуль `admin-order`, который владеет административным контуром заказов, дозаказов, payment events, refunds, financial holds, anti-fraud decisions, operator actions и audit trail. Feature #34 добавляет модуль `admin-service`, который владеет административным контуром сервисных кейсов, очередей, routing rules, SLA policies, сообщений, вложений, решений, refund/replacement/bonus actions, WMS events, supervisor override и audit trail. Feature #35 добавляет модуль `admin-identity`, который владеет административным master-data контуром пользователей, партнеров, сотрудников, sponsor relationships, eligibility rules, employee bindings, impersonation policies/sessions и audit trail.

## Backend modules and links
`auth` является владельцем runtime session state, role policies, invitation code state, active partner state, partner search audit и impersonation session. `admin-rbac` является владельцем административной модели внутренних доступов: учетные записи сотрудников, роли, permission sets, responsibility scopes, password/MFA/session policies, service accounts и audit events. `admin-cms` является владельцем административного CMS-контура: материалы, версии, blocks, SEO metadata, document metadata, review decisions, publication windows, relations и audit events. `admin-referral` является владельцем административного referral-контура: landing variants, landing versions, structured landing blocks, registration funnel versions, referral codes, attribution policy, attribution events, conversion events и audit events. `admin-pim` является владельцем административного PIM-контура: categories, brands, products, attributes, tags, media metadata, recommendations, merchandising blocks, import/export jobs и audit events. `admin-catalog` является владельцем административного catalog campaigns-контура: campaigns, issues, PDF/material metadata, pages, hotspots, freeze/rollover rules, archive projection, rollover idempotency и audit events. `admin-pricing` является владельцем административного pricing/promotions-контура: price lists, prices, promo prices, segment rules, promotions, shopping offers, offer items, gift rules, threshold benefits, pricing import/export jobs, publish idempotency и audit events. `admin-wms` является владельцем административного WMS/1C-контура: warehouses, stock snapshots, availability rules, supplies, supply lines, stock movements, reservations, sync runs, sync messages, discrepancies и audit events. `admin-order` является владельцем административного order management-контура: administrative order cards, supplementary orders, fulfillment groups, payment events, refunds, financial holds, risk events, operator actions, audit export и financial invariant checks. `admin-service` является владельцем administrative service management-контура: service cases, queues, routing rules, SLA policies, messages, attachments, decisions, refund/replacement/bonus actions, WMS events, supervisor board, quality review и audit export. `admin-identity` является владельцем administrative identity/master-data контура: subjects, masked profile attributes, partner relationships, office bindings, employee bindings, eligibility rules, impersonation policies/sessions и audit events. `platform-experience` является владельцем сквозного experience-state: consent preferences, notification preferences, runtime config snapshots, analytics adapter diagnostics и i18n missing key diagnostics. `employee` не нарушает владение существующих доменов: `order` остается владельцем базового customer order lifecycle, `cart` - корзины, `profile` - персональных и партнерских данных, `authorization` - identity/roles/MFA/locks, `partner-office` - поставок, payment/WMS/delivery/bonus системы остаются интеграционными источниками. Employee хранит support-контекст, ссылки, order-history read-model, employee claims read/write model, partner-card audit, partner-report snapshot, employee profile settings, elevated requests/sessions и audit trail действий сотрудников.

- `auth` вызывает `authorization` для проверки роли, route policy, MFA/elevated policy и блокировок.
- `auth` вызывает `employee` для проверки active elevatedSessionId при controlled impersonation.
- `auth` вызывает `admin-rbac` для получения session permission matrix и завершения затронутых сессий после block/deactivation.
- `auth` пишет sessions, roles, route policies, invitation code state, active partner, impersonation sessions и audit events в Postgres через Liquibase XML managed schema.
- `admin-rbac` вызывает `authorization` для role, permission set, scope и MFA/elevated policy checks.
- `admin-rbac` вызывает `auth` для чтения current session context и termination affected sessions при block/deactivation/emergency deactivation.
- `admin-rbac` пишет internal accounts, roles, permission sets, role bindings, account assignments, responsibility scopes, security policies, service accounts, service secret rotations и audit events в Postgres через Liquibase XML managed schema.
- `admin-rbac` ограничивает service account scopes для WMS/1C, payment, delivery и bonus integrations; протокол внешнего вызова остается у соответствующей интеграционной системы, RBAC выдает только технический scope и credential validation.
- `admin-cms` вызывает `authorization` и `admin-rbac` для проверки role, permission set, responsibility scope и elevated policy на каждом административном CMS endpoint-е.
- `admin-cms` пишет CMS materials, versions, content blocks, SEO metadata, relations, review decisions, publication windows и audit events в Postgres через Liquibase XML managed schema.

- `admin-cms` передает в `public-content` только published render model для главной, новостей, FAQ, документов, offer pages и home blocks; draft/review/archived versions не доступны публичному frontend.
- `admin-referral` вызывает `authorization` и `admin-rbac` для проверки role, permission set, responsibility scope и elevated policy на каждом административном referral endpoint-е.
- `admin-referral` пишет landing variants, landing versions, landing blocks, registration funnels, funnel versions, referral codes, attribution policies, attribution events, conversion events и audit events в Postgres через Liquibase XML managed schema.
- `admin-referral` передает в `public-content` только active benefit landing render model и campaign context; draft, paused, archived и future scheduled variants не доступны публичному frontend.
- `admin-referral` интегрируется с `partner-onboarding` по HTTP/JSON для validation referral code, сохранения campaign/referral context и sponsor attribution при регистрации feature #008.
- `admin-referral` передает `platform-experience` и frontend analytics layer только consent-aware conversion events и machine-readable metrics без секретов, токенов или лишних персональных данных.
- `admin-pim` вызывает `authorization` и `admin-rbac` для проверки role, permission set, responsibility scope и elevated policy на каждом административном PIM endpoint-е.
- `admin-pim` пишет categories, brands, products, product-category links, attributes, tags, media metadata, recommendations, merchandising blocks, import/export jobs и audit events в Postgres через Liquibase XML managed schema.
- `admin-pim` проверяет `fileReferenceId` изображений, PDF, сертификатов и вложений через S3/MinIO abstraction и не возвращает private storage paths во frontend.
- `admin-pim` передает в `catalog` только active categories и published product render data; draft, rejected, archived и товары без approved main image не доступны публичному frontend.
- `admin-pim` передает в `admin-cms` только опубликованные product/category references для product links и catalog links.
- `admin-catalog` вызывает `authorization` и `admin-rbac` для проверки role, permission set, responsibility scope и elevated policy на каждом административном endpoint-е.
- `admin-catalog` проверяет published PIM product data через `admin-pim`, чтобы hotspots не ссылались на draft, archived или audience-incompatible товары.
- `admin-catalog` проверяет `fileReferenceId`/storageKey PDF, обложек и page images через S3/MinIO abstraction и не возвращает private storage paths во frontend.
- `admin-catalog` передает в `catalog` только current published issue projection, PDF metadata, page images, hotspots и archive list; draft/scheduled/frozen выпуски не доступны публичному frontend.
- `admin-catalog` пишет campaigns, issues, material metadata, pages, hotspots, rollover jobs и audit events в Postgres через Liquibase XML managed schema.
- `admin-pricing` вызывает `authorization` и `admin-rbac` для проверки role, permission set, responsibility scope и elevated policy на каждом административном pricing/promotions endpoint-е.
- `admin-pricing` проверяет published PIM product data через `admin-pim`, чтобы price rows, bundle items и gift rules не ссылались на draft, archived или audience-incompatible SKU.

- `admin-wms` пишет warehouses, stock snapshots, availability rules, supplies, stock movements, reservations, sync runs/messages, discrepancies и audit events в Postgres через Liquibase XML managed schema.

- `admin-order` читает и обновляет административное представление заказа, supplementary order chain, fulfillment groups, payment events, refunds, financial holds, risk events, operator actions и audit events в Postgres через Liquibase XML managed schema.
- `admin-order` вызывает `admin-wms` по HTTP/JSON для warehouse context, reserve status, partial availability, shipment reference и blocked reason.
- `admin-order` вызывает `admin-pim`, `admin-catalog` и `admin-pricing` по HTTP/JSON для проверки published SKU, catalog period, price, offer, gift и bonus-affecting commercial rules при supplementary order.
- `admin-order` интегрируется с payment provider для capture/refund/dispute/chargeback protocol и обрабатывает callbacks идемпотентно по provider, externalPaymentId и idempotencyKey.
- `admin-order` передает employee module order support context, operator action results и audit links по HTTP/JSON, не передавая employee-модулю владение финансовыми переходами.
- `admin-order` использует bonus engine только для проверки согласованности bonus writeoff/compensation, а delivery integration - для shipment reference и delivery blocking context.
- `admin-service` вызывает `authorization` и `admin-rbac` для проверки role, permission set, queue scope, amount limit, supervisor override и elevated policy на каждом административном service endpoint-е.
- `admin-service` читает order/payment/refund context из `admin-order`, чтобы не дублировать финансовые суммы и не нарушать payment invariants.
- `admin-service` вызывает `admin-wms` по HTTP/JSON для return inspection, SKU availability и replacement shipment task.
- `admin-service` передает bonus compensation в bonus engine, а delivery integration использует для replacement delivery status и delivery issue context.
- `admin-service` проверяет вложения через S3/MinIO abstraction и не возвращает private storage paths во frontend.
- `admin-service` пишет service cases, queues, routing rules, SLA policies, messages, attachments, decisions, refund/replacement/bonus actions, WMS events и audit trail в Postgres через Liquibase XML managed schema.
- `admin-identity` вызывает `authorization` и `admin-rbac` для проверки role, permission sets, PII scope, responsibility scope, impersonation policy и elevated policy на каждом административном identity endpoint-е.
- `admin-identity` вызывает `auth` для завершения затронутых сессий при emergency block, проверки active impersonation context и запрета опасных действий в impersonation mode.
- `admin-identity` читает и обновляет masked profile projection через `profile`, проверяет sponsor/downline consistency через `mlm-structure`, employee bindings через `employee`, а order/offline sales eligibility через `admin-order`.
- `admin-identity` передает support subject context и impersonation audit links в `admin-service`, не передавая service-модулю владение master-data.
- `admin-identity` пишет subjects, profile attributes, partner relationships, office bindings, employee bindings, eligibility rules, impersonation policies/sessions и audit events в Postgres через Liquibase XML managed schema.
- `platform-experience` пишет consent preferences, notification preferences, runtime config snapshots, analytics diagnostics и i18n missing key diagnostics в Postgres через Liquibase XML managed schema.
- Frontend `AnalyticsProvider` отправляет consent-aware pageview/conversion events во внешние Yandex Metrika, Mindbox и Hybrid Pixel adapters; запрет consent блокирует отправку в соответствующий внешний канал.
- `employee` вызывает `authorization` для проверки роли, scope, MFA, блокировок и supervisor-доступа.
- `employee` читает partner profile/KPI из `profile` и `partner-reporting`, бонусный баланс и volume из `bonus-wallet`/bonus engine, order links и агрегированные статусы из `order`, payment event ids/statuses из payment, claim ids из claims, WMS batch/events из WMS/1C и delivery tracking из delivery.
- `employee` проверяет `fileReferenceId` employee-документов через S3/MinIO abstraction и не хранит приватные storage paths в API.
- `employee` передает `elevatedSessionId` и `correlationId` в связанные order, claim, partner, delivery и support flows как audit context для расширенных операций.
- `employee` пишет support/order-history/employee-claims/partner-card/profile/elevated read models и audit tables в Postgres через Liquibase XML managed schema.
